# NeuroGrim v4.0 publish process

> *develop → run gates → fix → re-run → publish*

This is the runbook for publishing a NeuroGrim release from v4.0
forward. Every release goes through the gate pipeline declared at
[`.claude/brain/publish-gates.yaml`](../.claude/brain/publish-gates.yaml)
before tagging.

The pipeline is defined by S12 (v4.0); the gate-by-gate semantics
live in `neurogrim explain publish-gates`. This doc focuses on the
operational sequence — what an operator actually types, in what
order, when shipping.

---

## Pre-flight (one-time, per developer machine)

Before your first publish:

```bash
# 1. Build the frontend bundle (RustEmbed snapshots it into the binary)
cd crates/neurogrim-dashboard/frontend
npm install
npm run build
cd ../../..

# 2. Build the binary (embeds the dist/ above)
cargo build --bin neurogrim

# 3. Install Playwright + Chromium (one-time; redirected to D: on
#    Windows when C: is under disk pressure)
cd crates/neurogrim-dashboard/frontend
PLAYWRIGHT_BROWSERS_PATH=/d/playwright-browsers npx playwright install chromium
cd ../../..
```

The `e2e-smoke` gate fails if Playwright isn't installed. The
`tests-pass` and `cargo-publish-dryrun` gates fail if the binary
isn't built.

## The publish flow

### 1. Develop on a feature branch

Standard git flow. Open PRs against `main`. The gates do NOT run
on every commit — they run before tagging.

### 2. Merge to `main` and prepare the release

```bash
git checkout main
git pull origin main

# Bump version in workspace Cargo.toml + each crate's metadata
# (rust-version, version, etc.). Run cargo's own check:
cargo build --workspace
```

### 3. Update CHANGELOG

Convert the `[Unreleased]` section to `[X.Y.Z] - YYYY-MM-DD`. The
`changelog-dated` gate enforces this — it greps for the
date-stamped header pattern and fails if it sees `[Unreleased]`
still floating.

### 4. Rebuild the frontend + binary

The `e2e-smoke` gate runs against whatever bundle is embedded in
the binary at build time. Out-of-date bundles cause confusing test
failures.

```bash
cd crates/neurogrim-dashboard/frontend && npm run build && cd ../../..
cargo build --bin neurogrim
```

### 5. Run the gate pipeline

```bash
# From the repo root
./neurogrim/target/debug/neurogrim publish-gate run --project-root .
```

What you see depends on your TTY:

- **Interactive (TTY)**: manual gates prompt y/N inline. Set
  `NEUROGRIM_OPERATOR=<your-handle>` in your shell first so inline
  ack writes the operator field.
- **Non-interactive (CI / piped)**: manual gates emit pending
  entries; ack via `neurogrim publish-gate ack --gate <id>` later.

Exit codes:

- **0** — every blocking gate passed → proceed to step 6.
- **1** — a blocking gate failed → fix the underlying issue, rebuild
  if necessary, re-run from step 5.
- **2** — a blocking gate is pending operator acknowledgement → ack
  it, then re-run (or use `--gate <id>` to run only the previously-
  failed gates).

### 6. Tag and push

Once `publish-gate run` exits 0:

```bash
git tag -a v<X.Y.Z> -m "v<X.Y.Z>"
git push origin v<X.Y.Z>
git push origin main
```

### 7. Publish to crates.io

```bash
cd neurogrim
# Order matters — dependencies first
cargo publish -p neurogrim-core
cargo publish -p neurogrim-sensory
cargo publish -p neurogrim-mcp
cargo publish -p neurogrim-a2a
cargo publish -p neurogrim-ecosystem
cargo publish -p neurogrim-dashboard
cargo publish -p neurogrim-cli
```

The `cargo-publish-dryrun` gate ran a `--dry-run` of every crate
in step 5, so the real publish should be no surprises. Watch for
crates.io rate limits (publish-with-30s-pause is conservative).

### 8. Bump submodule in ecosystem repo

```bash
cd /d/Brains
git submodule update --remote NeuroGrim
git add NeuroGrim
git commit -m "Bump NeuroGrim submodule to v<X.Y.Z>"
git push origin main
```

### 9. Restart production services

The 5 long-running A2A peers (8420 dashboard, 8421-8424 child
brains) need to pick up the new binary. The handoff convention is
"copy the local target/release/neurogrim.exe to
`D:\rust\cargo\bin\neurogrim.exe`" rather than `cargo install`
(avoids pressuring C:).

## When gates fail

### `doctor-clean`

The registry, culture, or publish-gates manifest has a structural
issue. Read the doctor output for the specific finding category.

### `tests-pass`

Workspace tests failing. Re-run with `neurogrim test --verbose` for
unfiltered cargo output, or `--retry-failed` to replay only the
prior batch.

### `cargo-publish-dryrun`

A crate can't be packaged. Common causes:
- A workspace-only file (e.g., a fixture in a parent directory)
  is referenced via a path crate doesn't ship.
- Version conflict between workspace declarations.
- Missing `description` / `license` in a crate's `Cargo.toml`.

Run the failing crate's dry-run alone for the exact error.

### `changelog-dated`

The `[Unreleased]` section hasn't been converted. Edit
`CHANGELOG.md` per step 3.

### `e2e-smoke`

Playwright failures usually mean either:
- The frontend or binary wasn't rebuilt before the run (the test
  binary serves stale UI).
- A genuine regression in the dashboard.

Re-run with `npx playwright test --headed` from
`crates/neurogrim-dashboard/frontend/` to see the browser, or open
the HTML report (`npm run e2e:report`).

### `review-changelog`

Manual. Operator hasn't ack'd yet. Either run interactively
(`--interactive` and answer y) or run the ack:
`neurogrim publish-gate ack --gate review-changelog`.

### `dashboard-renders-locally`

Manual. Same as above — ack via CLI after manually verifying.

## Adopter perspective

Adopters who don't have the NeuroGrim dashboard frontend (most v4.0
adopters) should:

- **Skip** the `e2e-smoke` gate type — declare your own `automated`
  gates with whatever browser tooling fits your stack.
- **Skip** the `dashboard-renders-locally` manual gate — replace
  with your own UI smoke checklist.
- **Keep** `doctor-clean`, `tests-pass`, `cargo-publish-dryrun`,
  `changelog-dated`, `review-changelog` — they're project-shape-
  agnostic.

The `neurogrim doctor` check for `publish-gates.yaml` is opt-in
during v4.0 rollout (silent when missing), so adopters can adopt
the pipeline at their own pace. Future stories may tighten this
once the v4.x ecosystem stabilizes.
