# Proposal §15.5 — `requires_neurogrim` version-pin convention

**Status:** DRAFT (authored 2026-05-01)
**Motivated by:** NeuroGrim IDE data-layer-plan-v4 §6.5 + round-3 §V11
**Blocks (in IDE):** Defense in depth — without this convention, a v5 substrate change could silently break the IDE
**Author:** NeuroGrim IDE team
**Reviewers:** LSP-Brains spec maintainers + future module-ecosystem authors

---

## TL;DR

Establish a convention for adopters and modules to declare their
substrate version compatibility:

```jsonc
// In a brain-registry.json or module-manifest.json:
"requires_neurogrim": ">=4.0.0,<5"
```

Three checks reinforce each other (defense in depth):

1. **Static metadata** — the field declares intent
2. **Runtime probe** — adopter calls `neurogrim --version` at startup
   and compares to its declared range
3. **SQLite schema-version handshake** — `MetricsHandle::open` already
   checks `metric_schema_version`; surface the mismatch as a structured
   error, not a panic

Round-3 critique §V11 named this "yes, both" — meaning all three
mechanisms together. This proposal makes that the spec convention.

---

## Motivation

### The silent-breakage scenario

Round-1 critique §VI laid this out:

> "NeuroGrim 4.1 ships an additive schema change (a new tag-stat column,
> say). The dashboard and CLI use the new schema. The IDE's open
> connection still has the old schema cached. Reads silently return
> wrong data, OR fail with `disk image is malformed`."

The IDE's data-layer-plan-v4 fixed this for `MetricsHandle` specifically
(open-and-close per operation; schema check at open time). But it
didn't fix the broader pattern: any v4.x change that breaks an
expectation the IDE has — the bus payload shape, registry-edit
endpoint, awareness file format — could silently break adopters.

### Why the IDE specifically wants this

The IDE links `neurogrim-core` as a Rust dep (Phase A v1) at version
`^4.0.0`. Cargo's semver resolves to whatever 4.x is current, which
might include semantic shifts the IDE didn't expect.

Beyond Cargo's coarse-grained protection:
- The IDE consumes `_neurogrim/notifications` payload shape — if v4.x
  silently changes it, useTopic deserializes wrong values
- The IDE assumes registry-edit is a 4-step process; if v4.x adds a
  fifth required step, the IDE's prototype-Path-A (per IDE plan §4.5)
  drifts
- The IDE assumes `LocalAwareness` has facts + notes; if v4.x adds a
  third collection (per future §15.x proposal), the IDE silently drops it

A version pin + runtime probe catches the gap loudly at startup
instead of silently mid-session.

### The case beyond IDE

Future adopters writing modules will need this same protection. A
`git-panel` module shipping against v4.x assumes certain capability
manifest fields exist; a v5 might restructure them. The version pin
lets the module say "I require neurogrim 4.x" and the runtime catches
the violation at install-time.

## Proposal

### 1. The field

For Brain registries (top-level adopters):

```jsonc
// In brain-registry.json
{
  "config": { ... },
  "meta": {
    "project": "neurogrim-ide",
    "schema_version": "2.1",
    "requires_neurogrim": ">=4.0.0,<5"   // NEW
  }
}
```

For module manifests (already in the IDE's manifest v2):

```jsonc
// In modules/<id>/module-manifest.json
{
  "$schema": "https://neurogrim.dev/schemas/module-manifest-v2.json",
  "id": "neurogrim-ide-editor",
  "version": "0.1.0",
  "requires_neurogrim": ">=4.0.0,<5",   // already in IDE manifest v2
  ...
}
```

Format follows Cargo's semver-comparator syntax (the well-known one):
- `>=4.0.0` — minimum
- `<5` — exclusive maximum
- Comma-separated AND
- Pipe-separated OR (`>=4.0.0,<5 || >=5.2.0,<6` — for adopters that
  support a range with a forced minor in v5)

### 2. The three reinforcing checks

#### Check 1: Static metadata

The field MUST be present in adopter brain-registries and module
manifests. v4.0.0 doesn't enforce this (no field-level validation
beyond schema parsing); v4.x optionally validates that the field is
present and well-formed.

Module installer (`neurogrim ide install <module>` for the IDE,
analogous CLI for future module-ecosystem-supporting adopters)
checks the field at install time and **refuses** to install a
module that requires a version of NeuroGrim that doesn't match the
running CLI.

#### Check 2: Runtime probe

At startup, the adopter calls `neurogrim --version` (already exists,
returns `neurogrim 4.0.0` in v4.0.0). Compares the result against
its declared `requires_neurogrim` range.

```rust
// Pseudocode for the IDE's startup
let cli_version = run_cli_version_probe()?;  // "4.0.0"
let required = self.brain_registry.requires_neurogrim()?;  // ">=4.0.0,<5"

if !semver_match(&cli_version, &required) {
    return Err(StartupError::IncompatibleNeurogrimVersion {
        cli_version,
        required,
        hint: "upgrade neurogrim CLI to a compatible 4.x version, or update brain-registry.json::meta.requires_neurogrim"
    });
}
```

Failure is loud (refuse to start) — silent limp-along is the bug
this proposal exists to prevent.

#### Check 3: SQLite schema-version handshake

`MetricsHandle::open` (per neurogrim-core) already inspects the
`metric_schema_version` table. When the IDE opens the handle, a schema
mismatch surfaces as `Err(Error::SchemaVersionMismatch { expected, found })`.

This proposal asks `neurogrim-core` to:
- Define analogous version checks for the bus topic format (the
  `messages` table or per-line JSONL format) and the LocalAwareness
  file format (`schema_version` field already exists in v4.0.0)
- Surface mismatches as structured errors, not panics, with `expected`
  and `found` fields the IDE can include in its startup error

### 3. Where errors land

When a version mismatch is detected at IDE startup:

- **Loud error UI** — full-screen "Substrate version mismatch" overlay
  with: declared range, actual version, what to do
- **No fallback to limp-along behavior** — refuse to render any
  panel, refuse to write to substrate, refuse to enter the operator's
  normal workflow
- **Recovery option** — "Edit brain-registry.json" button opens the
  file in a minimal text editor surface (the only operator action the
  IDE allows in this state); after edit, re-launch resolves the gap

This is intentionally hostile to "I'll fix it later" patterns — silent
data corruption from version drift is far worse than a startup-time
nag.

## Alternatives Considered

### A. No version pin; trust Cargo's semver resolution

Rejected — Cargo only protects the Rust crate dependency. Adopters
also depend on:
- The `neurogrim` CLI binary (separately versioned, separately
  installed)
- The on-disk file formats (`local-awareness.json`,
  `brain-registry.json`, `<topic>.jsonl`)
- The bus protocol shape

Cargo doesn't know about any of those. Per round-3 critique:
"defense in depth" applies because no single mechanism covers all
the surface areas where version drift could matter.

### B. Hard-coded version per adopter (no range)

`requires_neurogrim: "4.0.0"` exact. Rejected — too brittle.
Adopters should accept patch-level updates without re-pinning.

### C. Only check at install time, not at every startup

Cheaper at runtime. Rejected — the operator might upgrade the CLI
between IDE sessions; install-time check misses that.

### D. Spec the field but don't enforce

Adopters declare for documentation only; runtime never checks.
Rejected — that's exactly the silent-breakage scenario this proposal
is preventing. Documentation without enforcement is theater.

## Backwards Compatibility

| Concern | Impact |
|---|---|
| Existing brain-registries without the field | Treat as "no constraint" — runtime allows any neurogrim version. Field is optional in v4.x; adopters opt in. |
| Existing modules without the field | Same — treated as unconstrained. (Adopters who care, opt in.) |
| v4.0.0 CLI doesn't expose schema-version-mismatch errors | The third check (SQLite schema handshake) lands as v4.x; pre-v4.x checks rely on (1) and (2) only. |
| Multiple `requires_neurogrim` resolutions across modules | Each module checks independently; if the running CLI satisfies all of them, all install successfully. The IDE's startup checks the brain-registry's field; modules check theirs at install time. |

The proposal is opt-in at v4.x and becomes encouraged at v4.y once the
CLI and core both support all three checks.

## Acceptance Criteria

This proposal is accepted when:

1. **Spec text updated:**
   - [ ] LSP-Brains spec §15 (governance) defines `requires_neurogrim` field
   - [ ] Format documented as Cargo-style semver-comparator syntax
   - [ ] Three-check defense-in-depth rationale stated

2. **`neurogrim-core` ships:**
   - [ ] `version_constraint::matches(cli_version, required) -> bool` helper
   - [ ] Schema-version-check helpers for bus topic files and LocalAwareness
         (in addition to the existing `MetricsHandle` check)
   - [ ] Structured `SchemaVersionMismatch` error with `expected` / `found`

3. **`neurogrim-cli` ships:**
   - [ ] `neurogrim --version` continues to expose semver
   - [ ] `neurogrim doctor` includes a "version compatibility" check
         section (consume the field, validate, report)
   - [ ] (Optional) `neurogrim ide install <module>` validates module's
         `requires_neurogrim` before install

4. **NeuroGrim IDE:**
   - [ ] All 6 built-in manifests already have `requires_neurogrim:
         ">=4.0.0,<5"` (Phase C1 included this — no further IDE work needed)
   - [ ] IDE startup performs the runtime probe
   - [ ] Error UI for mismatch (full-screen overlay; refuse normal startup)

5. **Brain registries:**
   - [ ] Every NeuroGrim-managed Brain (NeuroGrim itself, the IDE,
         local-pc-operational-management children) updates
         `brain-registry.json::meta.requires_neurogrim` per their needs

## Open Questions

### Q1: What's the right default range for new adopters?

If the spec recommends `requires_neurogrim` but doesn't enforce it,
new adopter projects via `neurogrim init` should ship with a sensible
default.

**Recommendation:** `>=<current-version>,<<next-major>` — for v4.0.0,
that's `">=4.0.0,<5"`. `neurogrim init` writes this.

### Q2: How does this interact with the dev-time `[patch.crates-io]`?

The IDE uses `[patch.crates-io]` to point at local NeuroGrim source
during dev. The `cargo --version` from the patched crate might not
match the registry version. Does that count as a mismatch?

**Recommendation:** runtime probe uses `neurogrim --version` (the CLI
binary), not the linked crate version. Local crate patches don't
affect the CLI. So mismatch only triggers on real CLI version
differences. Document this in the proposal.

### Q3: What about cross-Brain federation with version-mismatched peers?

Brain A on v4.0.0 federates with Brain B on v4.1.0. A's registry pins
`<4.1`. B doesn't violate A's constraint (B is on its own version). Do
their A2A interactions check version compatibility?

**Recommendation:** out of scope for this proposal. Cross-Brain federation
is per IDE plan §4.9 / §7; if version mismatches in federation become
operationally relevant, future proposal handles. v1 of this proposal
just asks "is this Brain itself running on a compatible CLI?"

### Q4: Should the field also live on individual schemas?

A topic schema might require a specific neurogrim-core version (e.g.,
when `visibility` field per §15.3 ships). Worth a per-schema field?

**Recommendation:** out of scope for this proposal. Schema versioning
is §15.6 territory. If composability is needed, a follow-up proposal
can introduce per-schema `requires_neurogrim`.
