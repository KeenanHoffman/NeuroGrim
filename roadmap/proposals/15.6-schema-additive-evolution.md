# Proposal §15.6 — Schema additive-evolution enforcement

**Status:** DRAFT (authored 2026-05-01)
**Motivated by:** NeuroGrim IDE data-layer-plan-v4 §6.5 defense A10
**Blocks (in IDE):** A10 enforcement — modules can break consumers by changing their declared topic/series schemas
**Author:** NeuroGrim IDE team
**Reviewers:** LSP-Brains spec maintainers + adopters authoring schemas

---

## TL;DR

Enforce a single rule on schema evolution: **schemas may only widen
additively.** Specifically, between version N and N+1 of any schema
(topic payload, series, awareness namespace), changes are restricted
to:

- **Allowed:** add optional fields, relax constraints, expand enum
  values, weaken types (e.g., narrow → broader)
- **Forbidden:** rename, remove, narrow, change required-ness
  (optional → required), tighten types

Breaking changes require a new schema id (e.g., `git.events` →
`git.events.v2`).

The schema registry validates this at install/upgrade time and rejects
non-additive changes with a structured error.

---

## Motivation

### Round-2 critique attack A10

> "`git-panel` owns `git.events`. It can change the schema between
> releases. Subscribers depending on the old shape break. v4's
> schema-as-commitment principle says schemas can only widen additively
> (new optional fields), not narrow. The plan doesn't enforce this."

This is the same problem REST APIs and Protobuf face. The
LSP-Brains spec already states "schema as commitment" as a
methodology principle (per the IDE plan §2 design principles). This
proposal makes that principle **enforceable at install time**.

### Concrete IDE case

The IDE owns two adopter topics today:
- `neurogrim-ide/file-events` (payload: `kind`, `path`, `size_bytes`, `at`)
- `neurogrim-ide/session-events` (payload: `kind`, `value`, `at`)

If a future IDE version renames `kind` to `event_type` (a v1.5 → v2.0
shape change), every subscriber breaks silently — they deserialize
empty `kind` fields and either fail or get garbage data.

Today nothing prevents this break. The IDE could ship a new manifest
with the renamed schema and the runtime would happily load it. Only
when subscribers crash do operators notice.

### Why the spec needs to enforce, not just recommend

A "schemas should be additive" guideline that the spec just says-out-
loud is essentially documentation. The IDE plan §6.5 A10 wants
**runtime enforcement** — the module installer rejects schema changes
that violate the rule, surfacing the issue at install time.

For that to work cross-adopter, the rule needs to be in the spec, not
in each adopter's separate validator.

## Proposal

### 1. Schema versioning is mandatory

Every schema (topic, series, awareness namespace) declares a `version`
integer:

```jsonc
// modules/git-panel/schemas/git-events.topic.json
{
  "$schema":     "https://neurogrim.dev/schemas/bus-topic-v1.json",
  "id":          "git.events",
  "version":     1,                                // NEW: required
  "visibility":  "public",
  "owner":       "git-panel",
  "retention":   "transient",
  "payload":     { ... }
}
```

Versions are monotonic integers within a schema id. Going from v1 to v2
of the SAME id requires the changes between them to be additive (per
the rule below). To make a non-additive change, ship a new id (e.g.,
`git.events.v2-shape`) — by spec convention, the old id continues to
exist and old subscribers continue to work; new subscribers opt into
the new id.

### 2. The additive rule

**Allowed changes between versions of the same schema id:**

- Add a new optional field (`field?: T`)
- Add a new property to an enum
- Relax a constraint (e.g., `minLength: 5` → `minLength: 1`)
- Widen a type (e.g., `string` → `string | number`)
- Add a new value to a discriminated-union variant list
- Document something new (description changes always allowed)
- Bump expected_cadence_ms looser (`100ms` → `1000ms`)

**Forbidden changes between versions of the same schema id:**

- Rename a field
- Remove any field (even an optional one)
- Make an optional field required
- Narrow a constraint (e.g., `maxLength: 1000` → `maxLength: 100`)
- Narrow a type (e.g., `string | number` → `string`)
- Remove a value from a discriminated-union variant list
- Change owner module
- Change visibility (`public` → `private` is a break for existing subscribers)

When a forbidden change is detected, the rule violation is reported as
a structured error from the validator with the specific transition
that broke (e.g., `field 'kind' was required in v1 but optional in v2`).

### 3. The validator

`neurogrim-core::schema_evolution::validate_additive(prev, next) ->
Result<(), Vec<Violation>>` compares two schema versions and returns
all violations.

Where the validator runs:

- **At module install time** (`neurogrim ide install <module>`): check
  the new module's schemas against any installed version of those same
  schema ids. Reject install on violation.
- **At adopter publish time** (when an adopter writes a new schema
  version): the adopter's CI / `neurogrim doctor` runs the validator.
- **At schema-registry load time** (in the running IDE): if a v2 schema
  for an existing id is detected, validator runs. Mismatch → log
  finding + use the older version.

### 4. Backward-compat semantics

For a schema with version N and an existing N+1:
- Subscribers using the v1 schema continue to work — v2 is additive,
  v1's fields all still exist
- Subscribers updated to v2 can use the new fields

Publishers ALWAYS write the latest schema version. (Publishers can't
target older readers — additive evolution makes this safe; readers
ignore unknown fields.)

### 5. Discontinuous shape changes

When a non-additive change is genuinely needed (refactoring a topic's
purpose, normalizing field names, etc.), the discipline is:

1. Author a new schema id (e.g., `git.events` → `git.events.refactored`)
2. Both ids coexist for a deprecation period
3. New publishers write to the new id
4. Subscribers migrate when ready
5. After deprecation period, the old id is retired (removed from the
   schema registry)

The discontinuity is operator-visible (two schema ids in the catalog)
and gives consumers time to adapt.

## Alternatives Considered

### A. No enforcement; trust authors to follow the convention

Rejected — that's the status quo and doesn't solve A10. Authors do
make breaking changes without realizing.

### B. Major-version embedded in id (`git.events.v1`, `git.events.v2`)

Force every breaking change into a new id from day one. Rejected:
- Heavyweight for additive evolution (every minor version → new id)
- Doesn't actually prevent forbidden changes within a single id;
  authors could still ship `git.events.v1` v1 → v2 as a non-additive change

### C. Allow non-additive changes if marked `breaking: true`

Rejected — defeats the point. Authors mark "breaking" and consumers
don't see it until consumers crash. The hard rule is "you can't break
in place; ship a new id" with no escape hatch.

### D. Defer to JSON Schema's $version mechanism

JSON Schema 2020-12 has a `$version` field. Use it directly. Rejected:
- Doesn't apply to series schemas or awareness namespace schemas
  (those aren't JSON Schema documents in the strict sense)
- Doesn't define what "additive" means at the JSON Schema layer
- Different ecosystem; LSP-Brains needs its own clear rule

---

## Backwards Compatibility

| Concern | Impact |
|---|---|
| Existing v4.0.0 schemas don't have `version` field | Default to `1`. Adopter authoring a v2 must explicitly bump and run validator against v1. |
| Existing schemas might violate additive evolution if blindly bumped | Pre-acceptance state — no enforcement. Adopter authoring discipline prevents new breaks; old "informal" breaks aren't retroactively reverted. |
| The IDE's two existing topic schemas (`file-events`, `session-events`) | Each gains an explicit `version: 1`. Future evolution must be additive. |
| Cross-Brain federation with version-mismatched schemas | Subscriber runs whatever version it has. Per additive rule, older subscribers still parse newer messages (extra fields ignored). Newer subscribers parsing older messages: missing-optional handling already required by JSON deserializers. |

The proposal's discipline kicks in for new schemas authored after
acceptance. Old schemas grandfathered as v1 implicitly.

         `session-events.topic.json`) gain `version: 1`

## Open Questions

### Q1: How is "additive" defined for JSON Schema-shaped payloads?

JSON Schema is rich (refs, allOf, anyOf, etc.). The validator's
notion of "additive" needs to handle those. Specifically:

- Adding a field to `properties` (with `required: false`) — additive ✓
- Adding a field to `properties` AND `required` — forbidden ✗
- Adding to `oneOf` — additive (new variant accepted) ✓
- Removing from `oneOf` — forbidden (existing variants would be rejected) ✗
- Adding `additionalProperties: true` where it was `false` — additive ✓
- Adding `additionalProperties: false` where it was unspecified — narrowing, forbidden ✗

**Recommendation:** the validator implements a structural diff over
JSON Schema and applies the rule per node. v1 of this proposal handles
the common cases above; edge cases (recursive refs, complex pattern
properties) get logged as "validator can't decide" — operator manually
reviews.

### Q2: Series schema specifics

Series schemas have a different shape (kind, unit, description). What
counts as additive there?

- Adding optional `tags` allowlist → additive
- Changing `unit` (e.g., `ms` → `s`) → forbidden (semantic break;
  consumers see different magnitudes)
- Changing `kind` (`gauge` → `counter`) → forbidden (different
  aggregation semantics)
- Description changes → always allowed

**Recommendation:** spec these per the kind/unit/description structure.
v1 of the validator covers the listed cases.

### Q3: Awareness namespace schemas

Per the IDE plan §5, awareness namespace schemas declare key shapes
under a category. Additive evolution rules?

- Adding a new key spec → additive (new keys can be set; old code
  ignores them)
- Removing a key spec → forbidden (consumer code depending on the key
  breaks)
- Tightening a key's type → forbidden
- Loosening → additive

**Recommendation:** apply the same model. Validator handles awareness
namespace schemas same way.

### Q4: Cross-version subscriber/publisher mismatches

If a publisher is on v2 and a subscriber is on v1 of the same schema,
does the subscriber crash?

The whole point of additive evolution: NO. v1 fields all still exist
in v2; v1 subscriber sees them; new fields are unknown to v1
subscriber and harmlessly ignored (per JSON deserializer convention).

But this only works if consumers DO ignore unknown fields. Some
strict deserializers (e.g., Rust's `#[serde(deny_unknown_fields)]`)
fail-closed. The spec should recommend lenient deserialization on the
consumer side.

**Recommendation:** add a "consumer guidance" section to the spec
update that says: subscribers MUST tolerate unknown fields in
incoming messages. Strict-mode deserialization is incompatible with
additive evolution and should not be used for bus message consumers.
