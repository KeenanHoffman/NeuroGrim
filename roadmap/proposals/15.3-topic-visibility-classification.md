# Proposal §15.3 — Topic visibility classification

**Status:** DRAFT (authored 2026-05-01)
**Motivated by:** NeuroGrim IDE data-layer-plan-v4 §6.5 defense A9
**Blocks (in IDE):** A9 enforcement — modules can subscribe to other modules' topics regardless of intent
**Author:** NeuroGrim IDE team
**Reviewers:** LSP-Brains spec maintainers + adopters with multi-module ecosystems

---

## TL;DR

Add a `visibility` field to bus topic schemas. Three values:

```
visibility: "public"   // any module with bus.subscribe capability listing this topic can subscribe
visibility: "trusted"  // both publisher and subscriber must declare this topic in their manifests
visibility: "private"  // only the owning module can subscribe; fan-out happens but other modules see nothing
```

Default `public`. Schema-registered topics gain enforcement via the
capability checker; un-registered topics get the v4.0.0 status quo
(any subscriber sees everything).

The IDE's defense A9 in C4 wires the runtime check.

---

## Motivation

### Round-2 critique attack A9

> "A module declaring `bus.subscribe: ['git.events']` reads everything
> the git module publishes. There's no concept of topic-private-to-
> module-pair. A trojan module with subscribe-to-everything pattern
> harvests context from every other module."

Today (v4.0.0), bus topics are conceptually public — any subscriber
that knows the topic name and registers a listener gets every message.
This is fine for system topics like `_neurogrim/notifications` (they're
broadcast by design) but problematic for adopter topics that contain
context only the publisher and a specific consumer should share.

### Concrete IDE cases

The IDE owns two adopter topics today (`neurogrim-ide/file-events`,
`neurogrim-ide/session-events`). Both currently work fine as `public`
— they're broadcast for any consumer that wants them.

But Phase E of the IDE introduces topics where visibility matters:

- `neurogrim-ide/cell-hover-state` (intra-IDE coordination, not for
  external consumption) — should be `private`
- `neurogrim-ide/layout-events` — already published; could reasonably
  be `trusted` (federation peers can react, but third-party modules
  on the same Brain shouldn't snoop layout state)
- `git-panel/staging-events` (hypothetical future module) — wants to
  share staging-area state with the editor module but not with arbitrary
  monitoring extensions; `trusted` with the editor as declared subscriber

Without the visibility field, modules must either:
1. Trust every co-installed module (current state)
2. Encrypt payloads (heavy, wrong primitive)
3. Use OS-level sandboxing (out of scope for the bus)

### Why it's the spec's job, not the IDE's

The IDE could enforce visibility unilaterally for IDE-internal topics —
but the value compounds when the visibility classification is shared
across the ecosystem. A future `git-panel` module written by a third
party shouldn't have to invent its own visibility scheme; the spec
should define one.

## Proposal

### 1. Add `visibility` field to bus topic schemas

The IDE plan §5 already defines a `BusTopicSchema` shape including
`visibility`. This proposal makes that shape canonical:

```typescript
// Schema file: modules/<id>/schemas/<topic-name>.topic.json
{
  "$schema":     "https://neurogrim.dev/schemas/bus-topic-v1.json",
  "id":          "neurogrim-ide/file-events",
  "version":     1,
  "visibility":  "public" | "trusted" | "private",  // NEW (this proposal)
  "owner":       "neurogrim-ide-file-explorer",
  "retention":   "transient" | "audit" | "archive",
  "expected_cadence_ms": 1000,
  "description": "...",
  "payload":     { /* JSON Schema */ }
}
```

### 2. Define visibility semantics

```
public:
  Any module with `bus.subscribe.topics` declaring this topic name
  can subscribe + receive messages. The default for new topics.
  Equivalent to v4.0.0 behavior.

trusted:
  The publishing module's manifest must declare the topic in
  `bus.publish.topics` with `visibility: "trusted"`.
  Subscribing modules must ALSO declare the topic in
  `bus.subscribe.topics` with `visibility: "trusted"` (the visibility
  field surfaces in the subscriber declaration as a confirmation
  the consumer understands the trust requirement).
  Operator approves the publisher-subscriber pair at the subscribing
  module's install time.

private:
  Only the owning module (the publisher) can subscribe to its own
  topic. Fan-out continues to fire (if there's an SSE bridge or some
  other system-level reader, they still see messages), but no other
  module subscribing via the standard `bus_subscribe` IPC receives
  events. Effectively equivalent to "a module has its own internal
  pub-sub channel that happens to also persist to the bus."
```

### 3. Capability manifest interaction

The IDE manifest v2 (per IDE plan §6) already accepts:

```json
"bus.subscribe": {
  "topics":        ["neurogrim-ide/file-events"],
  "system_topics": []
}
```

This proposal extends to:

```json
"bus.subscribe": {
  "topics": [
    "neurogrim-ide/file-events",         // implicitly public
    {
      "topic":      "git-panel/staging-events",
      "visibility": "trusted"
    }
  ],
  "system_topics": []
}
```

The string form means "any visibility I'm allowed to read" (defaults
to public-or-trusted depending on the publisher's declaration).
The object form is operator-readable confirmation of trusted-tier
consent.

### 4. Capability checker enforcement

For each `bus.subscribe(topic)` call:

1. Look up the topic in the schema registry
2. If unregistered: treat as `public`, allow per existing rules
3. If registered:
   - `public` → allow if subscriber's manifest declares the topic
   - `trusted` → allow if BOTH publisher (in their schemas) and
     subscriber (in their bus.subscribe.topics) declare with
     `visibility: "trusted"` confirmation
   - `private` → allow only if the subscriber IS the owning module

This is what the IDE's defense A9 will implement once this proposal
lands and the `visibility` field is parseable.

### 5. Schema-registry-aware fan-out

Beyond capability check at subscribe time, the bus's fan-out path
itself can short-circuit `private` deliveries:

- For `private` topics, the broadcast loop checks subscriber-module-id
  against the topic's owner before delivering. Non-owner subscribers
  receive zero messages even if they somehow registered (defense in
  depth).

- For `trusted` topics, broadcasts deliver to all registered
  subscribers (capability check happened at subscribe time; runtime
  fan-out doesn't re-validate per-message for performance).

## Alternatives Considered

### A. ACL-based per-topic-per-module

Maintain a per-topic ACL of module ids that can subscribe. Rejected:
- Operationally heavy — operators manage ACLs per module install
- Doesn't scale across federation (each peer has its own modules)
- Less expressive than the three-tier model

### B. OS-level capabilities (e.g., per-process Linux namespaces)

Run each module in its own process with kernel-enforced isolation.
Rejected:
- Out of scope for the bus protocol
- Doesn't help in single-process deployments (the IDE in v4.x is
  one process)
- Doesn't compose with WASM-based future module sandboxing
  (different layer)

### C. Per-message recipient tagging

Publishers attach a recipients list to each message; the bus filters
delivery. Rejected:
- Forces publishers to know subscribers (breaks pub-sub decoupling)
- Per-message overhead vs schema-time classification

### D. Status quo (no visibility classification)

Keep v4.0.0 behavior; let adopters use ad-hoc per-topic conventions.
Rejected:
- The IDE specifically needs A9 enforcement for its threat model
- Other adopters will hit the same problem (round-2 mentioned
  ashlen-it's owned-children pattern as an example)
- "Each adopter solves it themselves" leads to fragmentation

## Backwards Compatibility

| Concern | Impact |
|---|---|
| Existing v4.0.0 topic schemas (no `visibility` field) | Treated as `public` — unchanged behavior |
| Existing `bus.subscribe` declarations | Unchanged — string form continues to work |
| Existing publisher behavior | None — visibility classification doesn't change publish semantics, just subscribe |
| `_neurogrim/*` system topics | Treated as `public` by default; the existing system-topic operator approval (IDE plan §6.5 A6) is the gate. Future spec text can promote specific system topics to `trusted` if needed (e.g., `_neurogrim/skill-invocations` → `trusted` because of the privacy concern raised in round-2 attack A6). |

The migration is gradual: schema authors can add `visibility` to their
topic schemas at any time; the runtime enforces classification only
for schemas that declare it.

   - [ ] LSP-Brains spec §16: bus-topic-schema definition includes
         `visibility` field with the three-value enum
   - [ ] Manifest §6: `bus.subscribe.topics` accepts both string and
         object forms

2. **`neurogrim-core` ships:**
   - [ ] `BusTopicVisibility` enum (`public | trusted | private`)
   - [ ] Schema parsing tolerates absent `visibility` (defaults to
         `public` per backwards-compat)
   - [ ] Optional helper: `BusState::publish` short-circuits delivery
         to non-owner subscribers for `private` topics

3. **Schema registry**:
   - [ ] Schema registry (per `neurogrim-ide`'s in-process registry, OR

         exposes a `lookup_visibility(topic)` API
   - [ ] Tests covering each visibility level's enforcement

4. **CLI:**
   - [ ] `neurogrim queue tail <topic>` works regardless of visibility
         (operator-level access, not module-level)
   - [ ] `neurogrim queue stats` includes per-visibility counts

5. **NeuroGrim IDE:**
   - [ ] Capability checker recognizes the new manifest form
   - [ ] Defense A9 in C4 enforces visibility at the IPC subscribe boundary
   - [ ] Existing `neurogrim-ide/file-events` topic stays `public`
         (no change); `neurogrim-ide/cell-hover-state` (Phase E) ships as `private`

## Open Questions

### Q1: Where does the schema registry live?

The IDE plan §5 has it in-process (loaded at module install).
That works for IDE-internal enforcement. But for cross-process
enforcement (e.g., the dashboard wanting to know visibility), a
shared registry helps.

**Recommendation:** keep the IDE's in-process registry as the
primary source of truth for IDE-loaded modules; expose a `neurogrim
queue schema show <topic>` CLI command that reads any Brain's
schemas from disk. The dashboard reads the same disk content.
Single canonical location: `<project>/modules/<id>/schemas/`.

### Q2: Does the publisher's manifest declare the topic's visibility, or does the schema?

Two ways to express "this topic is trusted":
- Schema file owns it (clean — schema is the source of truth)
- Publisher's manifest declares it on the publish side too (redundant)

**Recommendation:** schema owns it. Manifest's `bus.publish.topics` is
just a list of topic names the module is allowed to publish — visibility
classification belongs to the topic itself, not per-publisher.

### Q3: What about cross-Brain delivery via federation?

Federation reads of bus topics: do `private` topics travel?

**Recommendation:** `private` topics are owner-Brain-only (don't expose
via A2A). `trusted` topics expose only to enumerated peers per IDE
plan §4.9. `public` topics exposed to any peer that requests them.

This composes with §15.7's LocalAwareness visibility classification
— same three-tier model applied to a different primitive.

### Q4: Multi-publisher topics

What if two modules publish to the same topic name?

**Recommendation:** v1 of this proposal forbids it — topics have
exactly one declared owner via the schema's `owner` field. If a
shared-publish-topic use case emerges, a future proposal can extend.
