# Proposal §15.1 — Adopter metric-events topic convention

**Status:** DRAFT (authored 2026-05-01)
**Motivated by:** NeuroGrim IDE data-layer-plan-v4 §4.2 + §10
**Blocks (in IDE):** Phase E self-observability — the IDE's `ide.startup-ms`, `ide.command-surface-query-ms`, `ide.cell-bounces-per-min`, etc. cannot be ingested into the TSDB without this convention
**Author:** NeuroGrim IDE team
**Reviewers:** LSP-Brains spec maintainers + adopters wanting to record custom metrics

---

## TL;DR

Reserve the bus topic name `_neurogrim/metric-events` (system namespace,
auto-subscribed by `neurogrim-core`'s metrics ingest) for adopter-defined
metric points. Payload shape:

```json
{
  "series":   "<series-name>",   // adopter-or-system-namespaced; see §15.4
  "value":    <f64>,              // numeric value
  "ts_ms":    <i64>,              // unix milliseconds; absent → "now"
  "tags":     { "k": "v", ... }   // optional dimensions for query filtering
}
```

`neurogrim-core` ships a default subscriber that drains this topic and
calls `MetricsHandle::record` per message. Adopters publish metrics
**indirectly via the bus** rather than getting direct write access to
`metrics.sqlite` — preserves the round-1 critique's "single writer to
the TSDB" invariant.

---

## Motivation

### Round-1 critique's invariant

> "TSDB writes are a footgun. Your write process is no longer the only
> writer. NeuroGrim's `score` command writes here. The dashboard's bus
> subscriber writes here. The IDE wants to write here. Three writers
> contending on a single SQLite file."

The IDE's data-layer-plan-v4 took this seriously — §4.2 explicitly
removed any direct-write surface from the IDE's frontend. Adopter
metrics travel via the bus instead, so there's exactly one writer to
`metrics.sqlite` (the bus subscriber).

But the v4 plan **assumes** such a topic exists and is auto-ingested.
That assumption needs to be a spec commitment, not a hope.

### Concrete IDE need

The IDE's data-layer-plan-v4 §10 declares 7 series the IDE wants to
publish (excerpt):

| Series | Kind | Unit |
|---|---|---|
| `ide.startup-ms` | gauge | ms |
| `ide.command-surface-query-ms` | gauge | ms |
| `ide.cell-bounces-per-min` | counter | bounces |
| `ide.bus-subscribe-latency-ms` | gauge | ms |
| `ide.cmdb-read-latency-ms` | gauge | ms |
| `ide.federation-peers` | gauge | peers |
| `ide.module-active` | gauge | modules |

These power the `ide-performance` advisory domain (currently sitting at
score 50/100 — "honest unknown"). Without an ingest path, that domain
stays ungrounded indefinitely.

### Why this isn't already implicit

`neurogrim-core` already ingests `_neurogrim/score-snapshots` into the
TSDB (per the existing dashboard implementation). One could argue
adopters should "just" use that topic. But:
- `score-snapshots` is shaped specifically for unified Brain scores
  (`brain_id`, `unified_score`, `domain_breakdown`). Stuffing arbitrary
  `(series, value, ts_ms)` tuples into it is a misuse.
- Future tightening of `score-snapshots`'s shape (e.g., requiring a
  full domain breakdown) would break adopter-misused entries.
- Adopters need a clean affordance for "I have a metric to record"
  that doesn't require them to learn the score-snapshot domain.

A dedicated `_neurogrim/metric-events` topic with a minimal,
purpose-built payload solves this cleanly.

---

## Proposal

### 1. Reserve the topic name

Add to the LSP-Brains spec §16 (system topics list):

```
_neurogrim/metric-events
    Adopter-published metric points. Auto-ingested into the per-Brain
    TSDB at `_neurogrim/metrics.sqlite` by neurogrim-core's bus
    subscriber. Adopters use this instead of writing to metrics.sqlite
    directly — preserves single-writer invariant.
```

The system namespace prefix `_neurogrim/` matches existing reserved
topics (`_neurogrim/notifications`, `_neurogrim/config-changes`,
`_neurogrim/score-snapshots`).

### 2. Define the payload shape

```rust
// In neurogrim-core::queue (or a new `neurogrim-core::metrics_events` module)
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct MetricEventPayload {
    /// Series name. Per §15.4, adopter-defined series are
    /// `<adopter-id>.<name>` (e.g., `ide.startup-ms`). System series
    /// follow the existing flat-name convention (grandfathered).
    pub series: String,

    /// Numeric value. Series of `kind: counter` typically pass
    /// 1.0 (increment); gauges pass the absolute value.
    pub value: f64,

    /// Unix milliseconds. Absent / null → ingest at `now()`.
    #[serde(skip_serializing_if = "Option::is_none")]
    pub ts_ms: Option<i64>,

    /// Optional dimension tags. Mirrored into `MetricsStore::record_at`
    /// via the existing `Tags` type. Empty when no dimensions apply.
    #[serde(default)]
    pub tags: BTreeMap<String, String>,
}
```

### 3. Auto-ingest subscriber

`neurogrim-core` ships a service component (or the dashboard ships
it; see Open Questions) that:

1. Subscribes to `_neurogrim/metric-events` on every Brain it serves
2. For each message:
   - Validates the series name matches §15.4 namespace conventions
   - Validates value is finite (non-NaN, non-±∞) — defense A3 from
     IDE plan §6.5
   - Resolves `ts_ms` (absent → `now()`)
   - Calls `MetricsHandle::record_at(series, &tags, value, ts_ms)`
3. On any validation failure: emits a finding under the publishing
   adopter's domain (which adopter? the topic publish includes the
   message id; subscribers can correlate, but this is best-effort)
   and silently drops the message

### 4. Capability requirements

Adopters publishing metrics need `bus.publish: ["_neurogrim/metric-events"]`
in their manifest. Per existing `_neurogrim/*` reserved-namespace rule
(IDE plan §6.5 A5), this requires explicit operator approval at
install time — the operator confirms "this module wants to publish
metrics."

This intentionally makes metric-publishing a deliberate choice, not a
side-channel adopters can quietly tap.

## Alternatives Considered

### A. Direct `MetricsHandle::record` access for adopters

Expose write access to `metrics.sqlite` directly via a Rust API
adopters call from their own backend code. Rejected:
- Re-creates the multi-writer problem the round-1 critique killed
- Adopters that don't have a Rust backend (browser-only, JVM, etc.)
  can't use it
- No audit trail — bus events are inspectable post-hoc, direct
  writes vanish

### B. Per-adopter topic (`<adopter-id>/metric-events`)

Each adopter gets their own metric topic in their own namespace.
Rejected:
- Forces N subscribers (one per adopter) instead of one shared
- The auto-ingest subscriber would need a discovery mechanism to
  find adopter topics
- Adopters could already do this if they wanted ("publish to my own
  topic, write my own subscriber that records into metrics.sqlite")
  — but the per-adopter topic doesn't help if the goal is shared
  ingest infrastructure

### C. Use existing `_neurogrim/score-snapshots`

Stretch the score-snapshots payload to carry adopter metrics by
adding optional fields. Rejected per "Why this isn't already
implicit" above — misuses a purpose-built shape.

### D. HTTP endpoint (`POST /api/metrics/record`)

Expose a Plumbing endpoint adopters POST to. Rejected:
- Forces dashboard dependency for metric publishing
- IDE's data-layer-plan-v4 §4.5 specifically moved away from this
  pattern for registry edits (round-3 verdict: dashboard-coupling is
  fragile)
- Bus is the established system-event channel; consistency matters

## Backwards Compatibility

| Concern | Impact |
|---|---|
| Existing `_neurogrim/*` topics | None — additive new topic |
| Existing `MetricsHandle::record` callers (CLI `score`, dashboard) | None — direct callers continue to work; the bus subscriber is an additional path |
| Existing TSDB schema | None — same `record_at(name, tags, value)` API |
| Existing adopters | None — no current adopter publishes metrics; this proposal defines the future-shape |

Pre-acceptance state: the IDE has a `useSeries` hook (Phase B v2
shipped) that READS from the TSDB. It has no path to WRITE — Phase E
self-observability work blocked until this proposal lands.

## Acceptance Criteria

This proposal is accepted when:

1. **`neurogrim-core` ships:**
   - [ ] `MetricEventPayload` type
   - [ ] `_neurogrim/metric-events` reserved in the topic-name allowlist
   - [ ] `MetricsIngestService` (or equivalent) — bus subscriber that
         drains metric-events into `metrics.sqlite`
   - [ ] Validation: finite values, sane timestamps (±1h of now unless
         per-series `accept_backfill` is set per IDE plan §6.5 A3),
         series-name namespace check (§15.4)
   - [ ] Tests: round-trip publish→ingest→query

2. **Spec text updated:**
   - [ ] LSP-Brains spec §16 reserved-topics list includes `_neurogrim/metric-events`
   - [ ] Payload shape documented with example
   - [ ] Single-writer invariant for TSDB stated explicitly

3. **`neurogrim-cli` ships:**
   - [ ] `neurogrim queue tail _neurogrim/metric-events` works (already
         implicitly works via existing tail; verify shape renders well)
   - [ ] (Optional) `neurogrim metric record <series> <value>` CLI helper
         that publishes one event for ad-hoc use

4. **Auto-start of the ingest subscriber:**
   - [ ] Decision: lives in `neurogrim` CLI's `serve` mode, or in the
         `neurogrim-dashboard` background tasks, or both. See Open Q.
   - [ ] Whichever home: documented in spec; clear failure mode if not
         running (metrics published but never ingested)

5. **NeuroGrim IDE:**
   - [ ] Phase E ships `ide.*` series via this topic
   - [ ] `ide-performance` advisory domain starts producing real signal
   - [ ] Eventual promotion of `ide-performance` from advisory per
         spec §15.5 once series are stable

## Open Questions

### Q1: Where does the auto-ingest subscriber run?

Three candidates:

| Home | Pros | Cons |
|---|---|---|
| `neurogrim` CLI's hypothetical `serve` mode | Always-on, no UI dep | New CLI mode, new operational surface |
| `neurogrim-dashboard` (existing background tasks) | Existing infrastructure, consistent with score-snapshot ingest | Couples metrics publishing to dashboard being up — round-3 already flagged this concern for registry-edit |
| Adopter responsibility (each adopter spawns its own ingest helper) | Decoupled | Defeats the "shared infrastructure" point of this proposal |

**Recommendation:** start with dashboard hosting (matches existing
score-snapshot pattern) AND ship a `neurogrim core ingest` CLI
subcommand that adopters can run as an OS service when they don't
want dashboard dependency. v4.x ships both.

### Q2: How does the ingest service handle very late events?

If a metric event arrives with `ts_ms` from 5 days ago, do we ingest it?
- Yes → backfill works; useful for catching up after IDE was offline
- No → defends against poisoning (defense A3 territory)

**Recommendation:** per-series `accept_backfill` per IDE plan §6.5 A3.
Default reject if `ts_ms` is more than ±1h from now. Series that need
backfill (e.g., audit metrics) opt in via a future series-config
mechanism.

### Q3: Multi-Brain ingestion

When a Brain federates with another Brain, do their metric-events flow
to the federated peer's TSDB? Or stay local?

**Recommendation:** stay local. Each Brain's metric-events ingest into
its own `metrics.sqlite`. Cross-Brain queries happen via A2A read of
each peer's TSDB (the federation hook in IDE plan §4.9).

This keeps the single-writer invariant intact per Brain and matches
the round-2 critique's "per-Brain TSDB, not shared" verdict.

### Q4: Tag cardinality limits

Unbounded tag cardinality is a classic TSDB foot-gun (each unique tag
combination materializes as a series row). Should this proposal cap
distinct tag values per series?

**Recommendation:** out of scope for v1; document the foot-gun in
the spec and let adopters self-discipline. A future proposal can add
limits if real-world abuse appears.
