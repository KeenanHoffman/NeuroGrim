# Proposal §15.7 — Per-key LocalAwareness visibility classification

**Status:** DRAFT (authored 2026-05-01)
**Motivated by:** NeuroGrim IDE data-layer-plan-v4 §6.5 defense A11
**Blocks (in IDE):** A11 enforcement — federation reads of any LocalAwareness fact/note are blocked entirely until this proposal accepts
**Author:** NeuroGrim IDE team
**Reviewers:** LSP-Brains spec maintainers + adopters using federation

---

## TL;DR

Add a `visibility` field to `AwarenessFact` and `AwarenessNote` in
`neurogrim-core::awareness` (and to the corresponding LSP-Brains spec
section). One of three values:

```rust
pub enum AwarenessVisibility {
    Private,    // owner Brain only — never exposed via A2A
    Trusted,    // exposed to enumerated federation peers
    Public,     // exposed to any federation peer
}
```

Backed by a per-`AwarenessCategory` default (e.g., `ToolPaths` defaults
to `Private`; `OperatorPersonal` is reserved for `Private`-only). The
default makes adopting the proposal cheap — existing facts inherit a
sensible default — but lets fact authors override per-fact when they
know the data is safe to share.

---

## Motivation

### The IDE's threat scenario (round-2 critique attack A11)

> A federated peer's `LocalAwareness` is read via A2A by the IDE.
> Those facts can include things like:
> - cloud session tokens (`environment.aws_session_token`)
> - operator-personal preferences (`OperatorPersonal` notes)
> - environment-specific paths that reveal the layout of the operator's
>   filesystem
>
> A malicious extension running in the IDE with `federation.read.peers:
> ["any"]` and `scopes: ["any"]` can exfiltrate this. Worse: even
> well-intentioned modules don't know which keys are safe to expose,
> because the data model doesn't tell them.

This is a real exfil vector that exists today because `LocalAwareness`
is currently a single flat store with no mechanism to express "this fact
is fine to share" vs "this fact is a secret."

### Why the IDE can't ship A11 without this

The IDE's data-layer-plan-v4 §6.5 lists 12 defenses against capability
abuse. 11 of them landed (or have a clear path to land) in the IDE itself
during Phase C. **A11 is the one that requires upstream spec change.**

Until this proposal lands:
- The IDE's `useFederation()` hook hardcodes
  `federation.read.scopes: []` — empty list — which is interpreted by the
  capability checker as "no LocalAwareness reads from peers permitted."
- The federated cross-Brain queries the v4 plan envisioned (e.g., the
  command palette searching awareness facts across opened Brains) cannot
  ship safely.
- Operators federating Brains have no way to declare which of their
  facts are shareable, so the conservative behavior is to share none.

The IDE's data-layer-plan-v4 §13.7 (open question on federation read
defaults) and round-2 critique §V11 both flagged this gap. v3 of the
plan formally deferred A11 to this proposal; v4 added §15.7 to track it.

## Proposal

### 1. Add `AwarenessVisibility` enum to `neurogrim-core::awareness`

```rust
/// Federation-exposure classification for a fact or note.
///
/// Defaults are derived from the entry's `AwarenessCategory` (see
/// `default_visibility_for_category` below). Fact authors can override
/// per-fact when they know the data's exposure boundary differs from
/// the category default.
#[derive(Debug, Clone, Copy, Serialize, Deserialize, PartialEq)]
#[serde(rename_all = "snake_case")]
pub enum AwarenessVisibility {
    /// Owner Brain only. Never exposed via A2A federation reads.
    /// Suitable for: tokens, secrets, operator-personal data.
    Private,

    /// Exposed only to federation peers explicitly enumerated by the
    /// operator at peer-add time. Suitable for: machine-specific
    /// configuration that's safe to share within a trust boundary
    /// (operator's own machines, team-internal Brains).
    Trusted,

    /// Exposed to any federation peer that requests it. Suitable for:
    /// public technical facts (tool versions, OS family, hardware class)
    /// that have no secrecy concern.
    Public,
}
```

### 2. Add `visibility` field to `AwarenessFact` + `AwarenessNote`

```rust
pub struct AwarenessFact {
    pub key: String,
    pub value: String,
    #[serde(default)]
    pub category: AwarenessCategory,
    pub discovered_at: DateTime<Utc>,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub note: Option<String>,
    /// New in v4.x. Defaults to `default_visibility_for_category(category)`.
    /// `#[serde(default = ...)]` so existing v4.0.0 LocalAwareness files
    /// (which lack this field) deserialize cleanly.
    #[serde(default = "default_visibility_for_fact")]
    pub visibility: AwarenessVisibility,
}

pub struct AwarenessNote {
    pub content: String,
    #[serde(default)]
    pub category: AwarenessCategory,
    pub discovered_at: DateTime<Utc>,
    #[serde(default = "default_visibility_for_note")]
    pub visibility: AwarenessVisibility,
}
```

### 3. Per-category default

```rust
pub fn default_visibility_for_category(c: AwarenessCategory) -> AwarenessVisibility {
    use AwarenessCategory::*;
    match c {
        ToolPaths      => AwarenessVisibility::Private,    // paths reveal local FS structure
        Environment    => AwarenessVisibility::Private,    // could include shell, tokens
        OsQuirks       => AwarenessVisibility::Trusted,    // OS family is rarely secret but might be team-private
        Patterns       => AwarenessVisibility::Trusted,    // behavioral observations; team-shareable
        Constraints    => AwarenessVisibility::Trusted,    // hard constraints; team-shareable
        General        => AwarenessVisibility::Private,    // safe default for unknown content
    }
}
```

The "safe default" stance: when an adopter doesn't think about it, the
fact stays private. Marking something `Public` is an explicit, audit-
trailable choice.

### 4. Federation read enforcement contract

The `neurogrim-a2a` crate's `/api/awareness/...` endpoint MUST filter
returned facts/notes by visibility:

```
peer requesting via Trusted relationship → return Trusted + Public
peer requesting via Public relationship  → return Public only
non-federated read attempt              → 403 (already the case)
```

The "Trusted relationship" is determined by the local Brain's own
`brain-registry.json::config.federation.peers[<peer_id>].trust_level`
(defaulting to `Public` if not specified). The operator can elevate a
peer to `Trusted` at any time by editing the registry (autonomy-gated).

### 5. Migration path for existing `local-awareness.json` files

Existing v4.0.0 LocalAwareness files have no `visibility` field. The
serde default ensures they parse cleanly — every existing fact/note
gets the per-category default. **No migration tool required.**

Adopters who want to reclassify existing facts can edit the JSON
directly or use a future `neurogrim awareness reclassify <key> <visibility>`
CLI subcommand (out of scope for this proposal).

## Alternatives Considered

### A. No visibility field; per-peer scope tagging

Scope visibility at the peer level (some peers see all facts; others see
none) instead of per-fact. Rejected because:
- Operators rarely have a binary "trust this peer with everything" stance
- A single federation peer might legitimately need access to some facts
  (machine OS) but not others (cloud tokens)
- Doesn't compose with the v4 §6.5 capability model (which gates per-resource)

### B. Hardcoded per-category visibility, no per-fact override

Rely entirely on the category default; no per-fact field. Rejected:
- Some facts in nominally-private categories are actually safe to share
  (e.g., `ToolPaths` for a public open-source tool installed at a
  predictable path — `cargo` at `~/.cargo/bin/cargo` is rarely sensitive)
- Forces awkward category-stretching to express visibility intent

### C. Allowlist-based visibility (key-pattern → visibility)

A separate config file mapping key patterns to visibility levels.
Rejected:
- Splits the source of truth (the fact's home file vs an external rule)
- Hard to audit per-fact ("why is THIS fact marked private? where's the rule?")
- Doesn't survive `LocalAwareness` editing cleanly

### D. Encryption rather than visibility classification

Encrypt private facts at rest; let federation reads return ciphertext.
Rejected for this proposal:
- Heavier — adds key management as a new domain
- Doesn't solve the underlying classification problem (still need to
  decide what's private vs trusted vs public)
- Could be additive on top of visibility classification in a future
  proposal if encryption becomes a requirement

## Backwards Compatibility

| Concern | Impact | Mitigation |
|---|---|---|
| Existing v4.0.0 `local-awareness.json` files lack `visibility` field | None — serde default applies category-derived default | No-op for adopters who don't change anything |
| Existing v4.0.0 federation reads return all facts | **BREAKING for federation consumers** — after upgrade, peers see only facts whose visibility is at-or-below their trust level | Spec change applies only when both peers are on the new neurogrim-core version. Mixed-version federation: the older peer (sender) returns all facts as if visibility didn't exist; the newer peer (consumer) treats them as `Private` and ignores them. Conservative: lose data, never leak it. |
| Existing CLI `neurogrim awareness set <key> <value> [--category C]` | None — falls through to category default | New optional flag `--visibility` lands in 4.x.0 |
| Adopters writing facts via `LocalAwareness::upsert_fact` | None — current signature unchanged | New `upsert_fact_with_visibility` overload added; old fn forwards with category default |

The migration is gradual: any single Brain can adopt the new field (via
neurogrim-core upgrade) without breaking peers on older versions.
Federation consumers progressively gain filtering as their peer-set
upgrades.

## Acceptance Criteria

This proposal is accepted when:

1. **`neurogrim-core` ships:**
   - [ ] `AwarenessVisibility` enum with `Private`/`Trusted`/`Public`
   - [ ] `visibility` field on `AwarenessFact` and `AwarenessNote` with serde defaults
   - [ ] `default_visibility_for_category` function
   - [ ] `LocalAwareness::upsert_fact_with_visibility` (and old fn forwards)
   - [ ] Round-trip serialize/deserialize tests against v4.0.0 fixtures

2. **`neurogrim-a2a` ships filtered federation read:**
   - [ ] Federation read endpoint inspects peer's trust level
   - [ ] Filters returned facts + notes by visibility
   - [ ] Tests covering each (relationship × visibility) combination

3. **Spec text updated:**
   - [ ] LSP-Brains spec methodology: "Awareness visibility classification"
   - [ ] Schema doc: AwarenessFact/AwarenessNote v4.x shape

4. **CLI:**
   - [ ] `neurogrim awareness set <key> <value> [--visibility public|trusted|private]`
   - [ ] `neurogrim awareness list [--visibility-only public]` for auditing

5. **NeuroGrim IDE:**
   - [ ] `useFederation()` accepts non-empty `scopes: ["public"]` (and `["trusted"]` for enumerated peers)
   - [ ] Capability matcher unblocks federation.read.LocalAwareness when
         scopes is non-empty AND the spec change is detected (via
         `requires_neurogrim` pin per §15.5)
   - [ ] Defense A11 in C4 wires the runtime check

## Open Questions

1. **What's the right `Trusted` relationship granularity?**
   - Per-peer (each peer is independently trusted/public)?
   - Per-peer-pair (mutual trust required, like SSH known_hosts)?
   - Per-shared-realm (groups of peers trust each other, like a team's machines)?

   v1 of this proposal goes with per-peer for simplicity. Realm-based
   trust can be a future proposal if the per-peer model becomes
   operationally heavy.

2. **Should `Trusted` include the requesting peer's identity in the read response?**

   For audit purposes — an operator viewing their own LocalAwareness
   should be able to see "this fact has been read by peer X N times."
   Out of scope for this proposal but worth tracking.

3. **`OperatorPersonal` category proposal**

   Round-2 critique mentioned `OperatorPersonal` as a category. v4.0.0
   doesn't have it (the actual variants are `ToolPaths | Environment |
   Patterns | Constraints | General`). Should this proposal also add
   the `OperatorPersonal` variant? Worth: it makes the per-category
   default for "operator's preferred IDE etc." cleanly `Private`.
   Cost: enum surface grows.

   Recommendation: **add `OperatorPersonal` as part of this proposal**
   so the v4.x release has both the visibility model AND the category
   needed to express its most common use case.
