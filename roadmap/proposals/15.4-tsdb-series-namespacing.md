# Proposal §15.4 — TSDB adopter-series namespacing convention

**Status:** DRAFT (authored 2026-05-01)
**Motivated by:** NeuroGrim IDE data-layer-plan-v4 §4.2 + §10
**Blocks (in IDE):** Nothing hard-blocked — IDE can ship `ide.*` series under the proposed convention; this proposal makes the convention spec-blessed
**Author:** NeuroGrim IDE team
**Reviewers:** LSP-Brains spec maintainers + adopters authoring metrics

---

## TL;DR

Define forward-looking namespace conventions for TSDB series:

| Producer | Pattern | Example |
|---|---|---|
| System (NeuroGrim itself, going forward) | `_neurogrim/<name>` | `_neurogrim/score-snapshot-latency-ms` |
| Adopter | `<adopter-id>.<name>` | `ide.startup-ms` |
| Module within adopter | `<module-id>.<name>` | `git-panel.commits-per-day` |

**Existing flat names are grandfathered** — `brain_score`,
`request_duration_ms`, `cache_event`, `peer_probe_ms`, `bus_publish`,
`domain_score`, `domain_confidence` keep their current names. No
breaking change to v4.0.0.

---

## Motivation

### The accidental-collision problem

v4.0.0's TSDB has flat-named series: `brain_score`, `cache_event`,
`request_duration_ms`. These work fine as long as only NeuroGrim itself
writes to the TSDB.

When adopters start publishing custom metrics (per §15.1), name
collisions become likely:
- An adopter wants `cache_event` for their own thing — clobbers
  NeuroGrim's series semantically
- Two adopters both want `latency_ms` — neither can have it cleanly
- Future NeuroGrim work can't safely add new flat names without
  checking against every adopter's series

### Why the IDE needs this now

The IDE's data-layer-plan-v4 §10 declares 7 series the IDE wants to
publish:

```
ide.startup-ms
ide.command-surface-query-ms
ide.cell-bounces-per-min
ide.bus-subscribe-latency-ms
ide.cmdb-read-latency-ms
ide.federation-peers
ide.module-active
```

These already follow the convention this proposal defines (`<adopter-
id>.<name>`). The IDE could just ship them — but the convention should
be **explicit in spec**, not invented per-adopter.

### Round-1 critique on this point

> "v4's existing series (`request_duration_ms`, `cache_event`,
> `peer_probe_ms`, `bus_publish`, `domain_score`, `brain_score`,
> `domain_confidence`) are flat names with no prefix.
> The plan's `<adopter>.<series>` (dot-separated) is fine. It doesn't
> conflict because nothing exists.
> Recommendation: get this blessed in the LSP-Brains spec while you're
> at it. Propose:
> - `_neurogrim/<series>` for system series (the existing flat names
>   should arguably move under this)
> - `<adopter-id>.<series>` for adopter series (your `ide.*` plus
>   per-module `git.*`)
> This mirrors the bus topic reservation pattern and gives the same
> kind of namespace discipline."

Round-3 critique narrowed scope — existing flat names stay
grandfathered. This proposal adopts that narrowing.

---

## Proposal

### 1. Forward-looking convention

**System series** (added in v4.x and later):
```
_neurogrim/<lowercase-kebab>
```

**Adopter series:**
```
<adopter-id>.<lowercase-kebab>
```

**Module-within-adopter series:**
```
<module-id>.<lowercase-kebab>
```

Where:
- `<adopter-id>` matches the brain id in `brain-registry.json::meta.project`
- `<module-id>` is the module manifest's `id` field
- `<lowercase-kebab>` is lowercase ASCII alphanumeric + dashes

### 2. Grandfathering existing flat names

The following series are **explicitly grandfathered** as of v4.0.0:

```
brain_score
domain_score
domain_confidence
request_duration_ms
cache_event
peer_probe_ms
bus_publish
```

These keep their flat names indefinitely. Any future NeuroGrim work
that needs new system series uses the `_neurogrim/<name>` form.

(The grandfathered list is closed — it's exactly the v4.0.0 series. If
v4.x adds a new system series, it goes under `_neurogrim/` per the
forward convention, even if a flat name "would be cleaner.")

### 3. Validation at ingest

The metric-events ingest service (per §15.1) validates incoming series
names. Reject (and log a finding under the publishing adopter) if:

- Series name contains uppercase letters, underscores other than in
  grandfathered names, or characters outside `[a-z0-9.-/]`
- Series name has the `_neurogrim/` prefix but the publisher isn't
  NeuroGrim itself (defense: prevents adopters from squatting on system
  namespace)
- Series name has the `.` separator pattern but the prefix doesn't
  match the publishing adopter's id

This validation is informational, not blocking — invalid series still
get recorded but generate findings the operator sees.

### 4. Catalog support

`neurogrim-cli` ships:
- `neurogrim metric list` — list series with their producer (system /
  adopter / module) classification
- `neurogrim metric list --by-namespace` — group by namespace prefix
  for visibility into who's writing what

(The actual `series_catalog` table from the IDE plan v1 was dropped per
round-1 critique — v4.0.0's TSDB doesn't have one. If v4.x adds catalog
support, it integrates cleanly with this proposal's namespace
convention.)

## Alternatives Considered

### A. Force a migration to the new convention

Rename existing flat names to `_neurogrim/<name>` (e.g., `brain_score`
→ `_neurogrim/brain-score`). Rejected per round-3 critique:
> "Don't propose renaming existing `request_duration_ms`, `cache_event`,
> etc. — that's a breaking change."

Existing dashboards and queries depending on flat names would break.
The grandfathering compromise gives forward-looking discipline without
breaking the present.

### B. Reserve only `_neurogrim/` and let adopters use whatever else

A weaker version: spec reserves the `_neurogrim/` prefix, but doesn't
prescribe any structure for adopter series. Rejected:
- Doesn't solve the cross-adopter collision problem
- "Whatever you want" is a non-convention; spec text should be
  prescriptive enough to actually guide adopters

### C. Use `/`-separated names everywhere (uniform with bus topics)

Bus topics use `/` (e.g., `neurogrim-ide/file-events`). Why not series?

Rejected because:
- TSDB query backends (especially the existing rusqlite-based
  `MetricsHandle`) might treat `/` as a path separator — rusqlite
  itself doesn't, but tooling around the table might
- Dot-separated series names match Prometheus / Graphite / many other
  TSDB conventions, easing adopter familiarity
- Bus topics conceptually have hierarchical sub-paths; series names
  don't (they're flat strings with naming discipline imposed)

### D. UUIDs / opaque identifiers

Generate opaque series ids and let an external map track names.
Rejected — defeats the purpose of human-readable observability data.

---

## Backwards Compatibility

| Concern | Impact |
|---|---|
| Existing v4.0.0 series (the 7 grandfathered names) | None — preserved forever |
| Existing dashboards querying flat names | None — flat names continue to work |
| Existing TSDB schema | None — same `metric_points(name, tags, value, ts)` shape |
| New adopters publishing series | Land on the new convention from day one (or get a finding logged) |
| Future NeuroGrim system series | Use `_neurogrim/<name>` form |

This is a documentation and validation proposal, not a data migration.
Day-zero adopters of the new convention coexist with day-zero
NeuroGrim flat names without any disruption.

   - [ ] `neurogrim metric list --by-namespace` (or equivalent)
   - [ ] `neurogrim metric validate-name <series>` developer helper

4. **NeuroGrim IDE:**
   - [ ] `ide.*` series ship under this convention (already aligned)
   - [ ] No code change required for the convention itself; just publishes
         series with names matching the spec

## Open Questions

### Q1: What about modules whose id has a dot?

A module id like `git-panel-extra` is fine. But what if someone names a
module `git.panel`? Then `git.panel.commits-per-day` is ambiguous —
adopter `git` with module `panel`? Or adopter that-doesn't-exist with
module `git.panel`?

**Recommendation:** disallow `.` in module ids. Manifest validation
(C4 territory) rejects module ids matching `^[a-z][a-z0-9-]*$`.
That's already implicit in v4.0.0 conventions.

### Q2: What if an adopter id and a module id collide?

Adopter `git` and module `git` — can both exist? Not on the same
Brain (one would be the project, the other a module within it). On
different Brains, no collision because the metric-events publisher
includes the brain context via tags.

**Recommendation:** intentionally allow this. The discipline is per-
Brain; cross-Brain queries via federation already need brain-id tagging.

### Q3: How do federation peers report their series?

If Brain A federates with Brain B and queries `brain_score`, does B
return its own `brain_score` or A's? Both?

**Recommendation:** federation reads of TSDB return the federated peer's
own data. This already implicit in IDE plan §4.9. Cross-Brain
aggregation is application-level; the spec doesn't combine.

### Q4: Should this proposal include per-series tags namespacing?

Tags (Prometheus-style dimensions on a metric) are unbounded today. A
`brain_score` series can have any tag keys. Should the spec recommend
namespacing those too (e.g., `domain` is a system-reserved tag key)?

**Recommendation:** out of scope for this proposal. If tag-collision
problems emerge, future proposal addresses.
