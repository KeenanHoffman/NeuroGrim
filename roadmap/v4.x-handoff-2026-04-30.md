Detail for the highest-leverage pieces:

### Curated registry editors (`49fb3b1`)
- Sub-tab navigation on `/brains/:id/settings → Registry`: Weights | Autonomy | Hats | Federation.
- Single-Save semantics: the whole registry is one editable document; sub-tabs are projections; one Save validates + atomic-PUTs.
- ETag conflict detection from v1 carried forward.
- Deferred to v3: schemars-driven generic form generator; 3-way merge UI; domain definitions / `_todo_<name>` editor.

### Custom-page widget gallery (`953cdcd`)
- Reused v3.4 `LayoutEditorToolbar` with a new optional `pageId` prop. When set, PUTs to `/api/brains/:id/dashboard-pages/:pageId/layout`, hides the Reset button (no posture-aware default).
- New backend endpoint with the same shape as the legacy single-page PUT.
- Architectural choice: one widget editor for both Overview and custom pages — operators learn it once.

### Edit-via-bus keypath diffs (`ba3d02b`)
- New `crate::json_diff` module with `diff(before, after) → Vec<KeypathChange>`. RFC 6902-aligned ops (add/remove/replace), 100-entry cap.
- `emit_config_change_with_diff` extends the v1 emit helper with optional diff.
- Wired into `registry_put`, `dashboard_layout` PUT, and `dashboard-pages/:name/layout` PUT.
- Path format: `config.autonomy.action_types.edit-code.default_level` (dotted) and `widgets[2].size` (bracketed array indices).
- Subscribers can route by path prefix without re-parsing the registry.

### TLS auto-redirect (`e1470aa`)
- HTTP listener now auto-redirects GET on `/brains/<id>/secrets` to the HTTPS equivalent. Preserves Host header (localhost / 127.0.0.1 / LAN names all work) + query string. Returns 307 (not 301) so the redirect status follows the dashboard's actual config.
- Only redirects the page route, not the API. Agents hitting GET on the API for metadata keep working over HTTP; secret writes still get the existing 426 from S14-S-4.5 v3.

### Logs page (4 commits: `a0df6e9`, `a8cdf89`, `6756c2d`, `26b47f5`)
The Logs page was the largest single surface this session. End state:
- **6 sources wired**: publish-gates, approvals, invocations, notifications, score-history, services.jsonl. Each has its own filter chip with a count.
- **Per-row drill-down modal**: click any row → curated source-specific detail block + raw JSON fallback. Six per-source renderers.
- **Cross-page toasts**: `service_failed` SSE events surface a top-right toast regardless of which page the operator is on. Trigger policy is conservative — 7 silent events tested explicitly.
- **services.jsonl persistence**: new `crate::services::log_service_started/failed/stopped` helpers append to `<project>/.claude/brain/services.jsonl`. Best-effort writes (failures log via tracing but don't propagate). Wired into the start/stop/readiness-watcher paths.

### Per-peer log tail viewer (`1e9ba90`)
- New `crate::logs::read_peer_log_tail` reader: bounded read of trailing ~256 KB regardless of file size, drops partial first line when seeking mid-line, lossy UTF-8 decoding tolerates binary stderr.
- Endpoint: `GET /api/brains/:id/peers/:peer_name/log?lines=N`. Resolves peer via existing `resolve_peer`; reads from conventional `<peer_brain>/.claude/brain/logs/<peer>.log` so it works for peers started in previous dashboard sessions.
- Frontend: "View log" button per row on the Services page → centered modal with manual Refresh button + truncation hint.

### queue-config.yaml hot-reload (`5d362d0`)
- Closes a watcher-coverage inconsistency: every other config file already hot-reloaded; queue-config was missed.
- `BusState`'s `config: Arc<Option<QueueConfig>>` becomes `Arc<RwLock<Option<QueueConfig>>>`.
- `BusState::reload_from_path` re-reads YAML, swaps, clears backend cache. Parse errors preserve previous working config.
- Wired via `DashboardEvent::QueueConfigChanged` + a background subscriber task in `lib.rs::serve`.

## Architectural threads to follow (conventions established this session)

The next agent should match these patterns rather than invent new ones:

### 1. Centered modal pattern
SecretsPage's SetSecretModal, LogsPage's LogDetailModal, ServicesPage's
PeerLogModal all use the same shape:
- Fixed inset-0 backdrop with `bg-black/50`
- Centered panel with `max-w-{lg/2xl/4xl} m-4`
- Close affordances: × button, ESC key, backdrop click (with `e.stopPropagation()` on the panel)
- `role="dialog"` + `aria-modal="true"` + `aria-labelledby`

When you need a new modal, copy this shape — operators learn it once.

### 2. Per-source curated detail + raw payload fallback
Used in Logs drill-down and the peer-log truncation hint. Pragmatic
choice over per-source polish: a `DetailField` helper renders
`<label>|<value>` rows with stable `data-testid="detail-field-<label>"`
so tests can pin individual fields without coupling to layout. Raw
JSON in a collapsed `<details>` covers everything the curated block
doesn't.

### 3. TS exhaustiveness guards on event dispatchers
`AppShell.tsx::dispatchToastForEvent` uses:
```ts
const _exhaustive: Exclude<typeof event.kind, "service_failed"> = event.kind;
```
to force a compile error when a new `DashboardEvent` variant lands —
the next agent must consciously decide whether it should toast or
stay silent. Apply this pattern when adding new event-driven
dispatchers.

### 4. SSE-driven invalidation for everything with a backing file
Every config file the operator might edit gets:
1. A filesystem-watcher classification in `events.rs::classify_event`
2. A `DashboardEvent` variant in `events.rs`
3. A frontend invalidation rule in `useDashboardEvents.ts::invalidate`
4. (If it affects the bus) a backend reload subscriber in `lib.rs::serve`

The exhaustive list of watched files lives in `events.rs`. The
queue-config watcher coverage inconsistency this session closed shows
the value of keeping that list in sync.

### 5. Single-Save semantics for whole-document editors
The Registry editor's sub-tabs all share one Save button that
atomic-PUTs the full registry. Don't split into per-section saves
unless the document genuinely partitions (e.g., custom pages have
per-page layout PUTs because the URL scope partitions them).

### 6. Best-effort writes for audit ledgers
`crate::services::append_service_event` logs failures via tracing but
doesn't propagate them. Lifecycle operations must complete even if
the audit ledger can't be written. Don't let a degraded filesystem
cascade into operators being unable to start/stop their peers.

### 7. Bounded reads, never full-file
`read_peer_log_tail` caps at `PEER_LOG_TAIL_BYTES`. Any future
"read this potentially-unbounded file" endpoint should follow
the same pattern. Memory + response payload size must be O(1) in
file size.

### Polish that needs collaboration

2. **S15-C-9** — mobile-responsive breakpoints. Best paired with operator visual review at real 375px viewport. Mechanical pieces (no horizontal scroll, sidebar drawer at <768px, touch targets ≥44px) are doable solo, but priority calls need real-device feedback.

### Production polish — niche today

3. **S14-S-4.5 v5** — `tls-cert import <path>` for operator-supplied real-CA certs (production behind a reverse proxy). Private key stored via `SecretBackend` instead of a 0600 file (multi-user host deployments). The bundled cert lifecycle covers the dev-loopback case end-to-end without these.

### Substantial engineering (v5 candidates)

4. **Service restart recovery** — re-attach to peers that have a "started" event without a matching "stopped" by reading services.jsonl on dashboard startup. Tricky because there's no portable way to adopt a child process; "owning" a peer would require OS-level kill via PID + verifying agent-card before claiming the peer is the one we expected. Risk: if peer crashed and PID was reused, we'd kill the wrong process.

5. **S15-C-2 sensor refresh button** — trigger `neurogrim sensory <name>` for a peer to refresh its CMDB. Requires spawning child processes for arbitrary sensors, capturing output, surfacing results — its own engineering piece, not a tactical button.

6. **MCP ack-aware tool** — agents using SQLite-backed `ack_required: true` topics currently can't acknowledge messages from MCP tools, only via direct HTTP. Adding an ack tool to the MCP server would close this.

### Documentation pass (could ship anytime)

8. **Adopter walkthrough** — step-by-step "first custom-page authoring,

### Common gotchas

1. **`cargo test` on Windows occasionally hits "path too long"** — paths under `target/debug/deps/` exceed Windows' 260-char limit when the workspace is deep. Run from the workspace root, not `D:\Brains\`. If hit, `cargo clean -p <crate>` and retry.

2. **Frontend bundle is rust-embed'd** — changes to React components don't show up in the running dashboard until you rebuild BOTH `npm run build` (regenerates dist/) AND `cargo build -p neurogrim-cli` (re-embeds). Common confused-state mistake.

3. **ts-rs binding regen** — if you add a new type with `#[derive(TS)]`, the binding file at `crates/neurogrim-dashboard/bindings/<TypeName>.ts` is generated by the `cargo test` for that crate. Run `cargo test -p neurogrim-dashboard --lib export_bindings` to regenerate before TypeScript imports work.

4. **The frontend's `bigint` types from `u64` ts-rs derives** — `total_size_bytes: u64` becomes `bigint` in TS bindings. Coerce with `Number(value ?? 0)` when passing to UI helpers expecting `number`.

5. **Don't add `services_log_appended` toast triggers casually** — the dispatcher policy intentionally keeps `services.jsonl` events silent because `service_failed` already covers the actionable case. Adding more toasts risks alert fatigue. The TS exhaustiveness guard will catch additions; consider whether the new event genuinely meets the "operator-would-miss-this" bar.

## Decision log — choices the next agent should know about

These are non-obvious decisions made during this session that the
next agent might reconsider:

1. **C-4 v2 ships curated forms, not the schemars generator.** The
   spec preferred curated UX; the generator is harder than it looks
   (8+ days for the 80% case per plan-critic). Curated forms shipped
   for autonomy / hats / federation; v3 schemars is genuinely
   adopter-tail work. Don't reverse this without confirming an
   adopter actually needs the generator.

2. **Per-row Logs drill-down uses a centered modal, not a slide-in
   side sheet.** Modal blocks the timeline so it's clear focus has
   shifted. Side sheets allow comparison across entries but operators
   investigating one event at a time don't need that. If the user
   later asks for a side sheet, the existing modal pattern can be
   replaced — the LogEntry payload field is already there.

3. **Toasts only fire on `service_failed`.** Conservative trigger
   policy. The framework supports more triggers, but each new one
   should clear the "operator would miss this if not on the right
   page" bar. `notification_published` was the v1 candidate that got
   dropped because it requires a payload-driven severity convention
   that doesn't exist yet.

4. **Service restart recovery deferred to v5.** The "no Child handle
   for re-attached peers" problem is real engineering. Lite scope
   (just listing previously-running peers without re-attach) was
   considered but ultimately the value didn't clear the bar
   compared to alternatives. Worth revisiting if dashboard restart
   cycles become a frequent operator pain point.

5. **services.jsonl persistence shipped, even though restart
   recovery deferred.** The persistence is useful on its own (Logs
   timeline gains the 6th source; durable audit trail across
   restarts). Restart recovery would build on it but isn't a
   prerequisite for the Logs source.

6. **TLS auto-redirect targets only the page route, not the API.**
   Agents hitting GET on the API for metadata keep working over
   HTTP; redirecting them might break consumers that don't follow
   redirects. The page (human-facing) auto-upgrades; the API
   (agent-facing) keeps its v3 semantics.

7. **Hot-reload uses RwLock, not lock-free swap.** Performance
   concern is hypothetical (config reads are infrequent — they only
   fire on backend cache misses). RwLock is simpler and matches the
   existing `senders` / `backends` pattern.

## What to NOT do without asking the user first

- **Don't run `git push --force` or amend pushed commits.** Standard
  guard.
- **Don't reverse architectural decisions in the "Decision log"
  above** without explicit user confirmation. Each was made
  deliberately.
- **Don't add new toast triggers** without checking the "operator
  would miss this" criterion. Toast fatigue is the primary risk.
- **Don't introduce new frontend dependencies** without checking
  `package.json` first — the dashboard has been deliberately
  minimal-dep (no toast library, no form library, no chart library
  except recharts). Hand-rolled UI primitives are the convention.
- **Don't expand `data-testid` conventions arbitrarily.** Existing
  tests pin specific testids; renaming breaks tests across the
  codebase. New testids are fine; renaming established ones requires
  test updates.
- **Don't modify `domains/laas/`** — it's archived first-customer
  reference. Read-only by convention (per `CLAUDE.md`).
