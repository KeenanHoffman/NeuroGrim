## Why we haven't tagged yet

On 2026-04-23, a PyPI supply-chain incident surfaced publicly: a
**second-order, scanner-chain compromise**. A trojanized
security-scanner binary (the kind of tool that ironically exists to
*protect* against supply-chain attacks) captured CI credentials
during normal pipeline runs; those credentials were then used to
publish trojanized releases of an otherwise-legitimate package
whose maintainers were themselves the victims of credential theft.
The published releases carried a base64-encoded payload that
executed on `import` — a fork-bomb class of denial-of-service, but
the *shape* of the attack (embedded payload, pre-signature-
verification code paths, credentials-first escalation) is what
matters.

Our response is structural, not cosmetic:

- **No `cargo publish` runs** until NeuroGrim's own transitive
  dependency graph is clean by our own auditing, AND we ship a
  three-layer supply-chain awareness feature that gives adopters
  the same tooling.
- **Python SDK is dogfood-only**, not a shipped artifact —
  canonical SDK for downstream extension is Rust. See
  [`docs/sdk.md`](../sdk.md).
- **Our own security tooling is not on the trust chain we reject.**
  We do not ship, shell out to, or depend on Trivy or similar
  scanner binaries of the class that was compromised. Our Layer 1
  SCA is native-Rust, queries OSV.dev directly, and has a small,
  pinned, auditable library surface.

The eleven-epic scaffolding at `audit/ROLLBACK-PLAYBOOK.md` +
`~/.claude/plans/parallel-hugging-eich.md` documents the full
scope. Phase 0 (a self-audit baseline of the current codebase)
already shipped and remediated one active advisory (`rustls-
webpki` RUSTSEC-2026-0104, same-minor patch) — details at
`audit/phase0-report-2026-04-24.md`.

## Why 3.0, and why `rc.1`?

The workspace `Cargo.toml` default was `0.1.0` because no prior version
was ever published to crates.io or PyPI. The jump to `3.0.0-rc.1`
reflects **methodology maturity** — stages S1 through S10 of the
roadmap are shipped or infrastructure-complete — not a tagged-release
history.

`rc.1` (not `3.0.0`) because there are real gates still open. Those
gates are documented below rather than hidden. The RC designation buys
us the option to land fixes before `3.0.0` final without breaking
semver expectations.

## What you get

### Rust reference implementation

A five-crate Cargo workspace under `neurogrim/crates/`:

- **`neurogrim-core`** — pure scoring logic, zero I/O. This is the
  Brain engine.
- **`neurogrim-sensory`** — 12 built-in sensor domains (see list below).
- **`neurogrim-mcp`** — MCP server + client. `neurogrim serve` exposes
  seven BrainServer tools to Claude Code sessions (~983 tokens at
  session start).
- **`neurogrim-a2a`** — A2A peer protocol (spec §13): agent card,
  envelope, task client/server. Fractal composition and dual-brain
  topologies verified end-to-end.
- **`neurogrim-cli`** — the `neurogrim` binary.

`cargo test --workspace --all-targets` is green.

### 12 sensor domains

| Domain | Observes |
|---|---|
| `git-health` | commit discipline, branch state, tag currency |
| `test-health` | tests exist, pass, cover what they claim |
| `code-quality` | lint + type-check + complexity |
| `deploy-readiness` | CI config, Dockerfile, deploy scripts |
| `security-standards` | SECURITY.md, Dependabot, CODEOWNERS, SOC2/ISO27001/NIST evidence |
| `coherence` | cross-domain correlations fire appropriately |
| `human-comms` | brevity + calibration + per-hat overrides |
| `secret-refs` | safe credential catalog; no inline secrets |
| `docker-topology` | container health + A2A port layout |
| `agent-behavior` | LLM-as-judge verification of recommendation quality |
| `skill-coherence` | skill inventory matches current repo concerns |
| `capability-hygiene` | skill authoring quality vs a 1,536-char budget |

### Dual tool-invocation modes

- **MCP server** (`neurogrim serve`) — the default; seven scoring
  tools injected into the session. ~983 tokens at session start.
- **CLI mode** — zero-token mode for context-constrained sessions.
  Opt in by omitting NeuroGrim from `.claude/.mcp.json` and loading
  the `cli-mode` skill.

### Skill + hat system

- **20 plugin-format skills** under `.claude/skills/<name>/SKILL.md`
  with YAML frontmatter. `capability-hygiene` scores authoring quality
  against a declared character budget.
- **7 hats** (adversary, architect, incident-commander, rubber-duck,
  security-auditor, visionary, source-reader) as attentional lenses.

### Cultural substrate

A five-value invariant — positivity, integrity, honesty,
critical-but-kind, respect — declared in `culture.yaml`. Four
byte-identical copies across peer Brains; the ecosystem-level
`culture-coherence` domain flags drift.

### Governance infrastructure

- **Axis 4 v1 invocation ledger.** PostToolUse hook captures every
  `Skill` invocation (name + timestamp only — privacy by design).
  `capability-hygiene` classifies each skill as alive / dead / new
  against a 30-day grace period.
- **Gated domain promotion.** `abv-run promote` / `rollback` /
  `promotion-watch` with append-only `domain-promotion-ledger.jsonl`,
  three rebalance strategies, and an `ABV_OPERATOR` guard.
- **Judge-integrity ledger.** Red-sample calibration gate with triage
  CLI (`abv-run judge-integrity list | triage`).
- **Red-mode sweeps.** 13-sample, 6-scenario failure-mode library for
  adversarial validation.

### LSP Brains spec v2.5

- 15 normative sections + 7 appendices.
- Covers Brain architecture, registry schema, CMDB envelope, scoring
  model, correlation engine, MCP + A2A protocols, agent-behavior
  verification, and the domain promotion path (§15.5).
- Companion `METHODOLOGY-EVOLUTION.md` with 14 discovery-log entries.

### Peer + adoption topology

- **A2A peer protocol.** Agent card + envelope + task client/server.
  Fractal composition (parent↔child) and dual-brain (local↔external)
  both demonstrated across the four-Brain ecosystem.
- **Ecosystem Brain.** Six advisory domains visible only at the
  meeting point of children (spec-impl-alignment, terminology-
  coherence, protocol-boundary, north-star-alignment, ecosystem-
  trajectory, culture-coherence).
- **Python starter template.** A child Brain with 4 advisory domains,
  demonstrating the adoption pattern.

### Adoption surface (new in this release)

- **`docs/getting-started.md`** — ~20-minute path from clone to
  working Brain.
- **`examples/hello-brain/`** — minimal standalone demo with a
  pre-configured `brain-registry.json`.
- Top-level `LICENSE` files across all three Git repos.
- READMEs with a `🚀 Getting started in ~20 minutes` entry-ramp above
  the fold.
- Whitepaper §11 "Evidence Posture (2026-04-23)" — named primary
  hypothesis + bounded secondary measurement surface.

### Experimental evidence base

- 432-row `comparison-ledger.jsonl` from the 2026-04-22/23
  brain-vs-control experiment (Phases 1–3, plus 22-task held-out
  set). All pre-registered; falsification criteria locked before
  analysis; **kill decisions honored** — the held-out contradiction
  killed a tighter "factual-augmentation only" reframe on the
  `reframe/factual-augmentation` branch and stayed there rather than
  ship in an overstated form. Reports, ledgers, and post-mortem at
  `.claude/experiments/brain-vs-control/`.
- **Evidence + Hypothesis posture** (ROADMAP + METH-EV §14 +
  whitepaper §11): longitudinal value is the primary hypothesis;
  single-turn benchmarks are bounded instruments. Single-turn
  experiments cannot answer longitudinal questions — they were never
  built to.

## Recent experimental evidence

The 2026-04-23 brain-vs-control experiment answered several sub-
questions with pre-registered discipline:

- **L1 helps broadly on Sonnet**, not narrowly on a "factual-lookup
  class" — the held-out 22-task set killed the sharper reframe. See
  [`reframe-post-mortem.md`](../../.claude/experiments/brain-vs-control/reports/reframe-post-mortem.md).
- **Agent self-routing works** at the tool-invocation level (L2
  Phase 3). When given a `brain_query` tool with per-domain cost
  warnings, Sonnet routes ~100% on repo-aware tasks and 0% on
  trivial tasks.
- **Content freshness is load-bearing.** Stale or wrong injected
  content produces worse L1 outcomes than no context at all on
  factual-lookup tasks.
- **Haiku suffers from over-injecting.** Phase 1 pilot showed
  consistent regression; Sonnet+ is the recommended model for
  Brain-augmented sessions.

These are single-turn findings and are useful for tuning "what to
inject for this response." They are **not** tests of the Brain's
longitudinal value — see whitepaper §11 for the split.

## Adoption path

Pick your starting point:

1. **"I want to see a Brain produce a score."** — Start at
   [`docs/getting-started.md`](../getting-started.md); run
   `neurogrim score --project-root .` from inside
   [`examples/hello-brain/`](../../examples/hello-brain/).
2. **"I want to add a Brain to my Python project."** — Fork
   [NeuroGrim-python-starter](https://github.com/KeenanHoffman/NeuroGrim-python-starter).
   Install the SDK from source: `pip install -e
   ../NeuroGrim/sdk-python/`.
3. **"I want to build my own implementation in some other language."**
   — Read [LSP-Brains/INTRO.md](https://github.com/KeenanHoffman/LSP-Brains/blob/main/INTRO.md)
   (5 pages), then the spec.
4. **"I want to understand the whole thing."** — The whitepaper at
   [`whitepaper/WHITEPAPER.md`](../../whitepaper/WHITEPAPER.md) is the
   long-form — and §11 is the honest evidence posture.
