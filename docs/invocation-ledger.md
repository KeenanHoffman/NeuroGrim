# Invocation Ledger — Setup Guide

The invocation ledger captures every `Skill` tool call in a Claude Code
session and persists a one-line JSONL record per invocation to the
Brain's runtime state. The `capability-hygiene` domain reads the ledger
and classifies each skill as **alive**, **dead**, or **new** — closing
the empirical half of the three-axis self-observability picture
(hygiene · invocation · miss rate).

Axis 4 v1 (2026-04-22). Skills only. Subagents + tools are a future
extension once the v1 signal is validated in dogfood.

## What gets captured (privacy-by-design)

Each invocation records:
- `schema_version`: `"1"`
- `ts`: ISO 8601 UTC timestamp (generated by the hook, not Claude Code)
- `type`: `"skill"`
- `name`: the skill name (e.g., `rubber-duck`)
- `session_id`: Claude Code session UUID (for correlation only)
- `invocation_id`: the `tool_use_id` (unique per call)

**NOT captured:** tool arguments, tool responses, transcript content,
user prompts, agent reasoning. The v1 invariant is name + timestamp
only. Nothing sensitive enters the ledger; no scrubbing infrastructure
is needed.

## Installation

1. **Confirm the hook script exists.** This repo ships
   `scripts/record-skill-invocation.sh`. It's a shell script using
   only bash builtins — no `jq`, no Python, no external tools.

2. **Add the PostToolUse hook to your local settings.** The hook
   lives in `.claude/settings.local.json` (operator-local, gitignored)
   at the **project root where you run Claude Code**. For the
   Brains ecosystem that's `D:/Brains/.claude/settings.local.json`.

   Merge this into the file, preserving any existing `permissions`
   or other sections:

   ```json
   {
     "hooks": {
       "PostToolUse": [
         {
           "matcher": "Skill",
           "hooks": [
             {
               "type": "command",
               "command": "bash \"$CLAUDE_PROJECT_DIR/NeuroGrim/scripts/record-skill-invocation.sh\""
             }
           ]
         }
       ]
     }
   }
   ```

   Adjust the path if your Claude Code session is rooted somewhere
   other than the ecosystem root. For a NeuroGrim-only session, use:
   ```
   "command": "bash \"$CLAUDE_PROJECT_DIR/scripts/record-skill-invocation.sh\""
   ```

3. **Restart Claude Code.** Settings are read at session start; a
   restart ensures the hook is active.

4. **Verify the hook fires.** Invoke any skill (e.g., via
   `/keybindings-help` or any other harness skill). Check that
   `.claude/brain/invocation-ledger.jsonl` now has a line. Parsed
   JSON should look like:

   ```json
   {"schema_version":"1","ts":"2026-04-22T11:20:43Z","type":"skill","name":"keybindings-help","session_id":"...","invocation_id":"..."}
   ```

## Consumed by `capability-hygiene`

On every `neurogrim sensory capability-hygiene` run, the sensor reads
the ledger and adds per-skill fields:

- `usage_status`: `alive` | `dead` | `new`
- `usage_rarity`: `common` (default) | `rare` (declared in skill
  frontmatter)
- `window_days`: 90 (common) or 365 (rare)
- `invocations_in_window`: count in the relevant window
- `last_invoked`: most-recent timestamp (RFC 3339) or `null`
- `skill_age_days`: file mtime age in days

Aggregate counts in `capability_breakdown.skills`:

- `usage:alive`, `usage:dead`, `usage:new` — per-status totals
- `ledger:total_invocations` — overall ledger size
- `ledger:low_confidence` — `1` if below
  `LOW_CONFIDENCE_TOTAL_INVOCATIONS` (20), `0` otherwise

## Dead-skill findings

Dead findings are **advisory** — they don't subtract points from the
hygiene score (hygiene measures description quality; usage is a
separate axis). They surface in the `findings` array as:

```
dead-skill:<name>                      — normal signal
low-confidence-dead-skill:<name>       — ledger too sparse to trust
```

**Plan-critic discipline for acting on dead findings:**

- Combine with hygiene score before acting:
  - *dead + hygiene < 60* → probably misdescribed (fix the lead
    paragraph before declaring the skill retired).
  - *dead + hygiene ≥ 90* → probably genuine niche (consider
    `usage-rarity: rare` or archive).
- Wait for `ledger:total_invocations` to exceed 20 before treating
  signals as firm. Below that, every finding is marked
  `low-confidence-*`.
- A file freshly created (mtime < 30 days) is `new`, not `dead` —
  grace period honors skills that haven't had time to accrue
  invocations.

## Troubleshooting

**Hook doesn't appear to fire** — Check:
1. `.claude/settings.local.json` syntax is valid JSON.
2. Path to the script is correct and the file is executable
   (`chmod +x`).
3. On Windows, Claude Code uses Git Bash — the shebang `#!/bin/bash`
   should work. `/dev/stdin` is NOT available on Git Bash (use `cat`
   without arguments to read stdin).
4. Claude Code was restarted after the settings edit.
5. You're in a fresh session — existing sessions may have cached
   settings.

**Ledger lines are malformed** — The capability-hygiene reader
tolerates malformed lines (silently skips them). If this is
systematic, check the hook script — the printf format string has to
escape quotes correctly.

**CMDB shows all skills as "new"** — This is expected right after a
git clone (all file mtimes are the clone time, so all skills are
inside the 30-day grace period). Within a month, skills will either
accrue invocations or age out of the grace period and start getting
flagged dead.

## Cross-Brain behavior

Each Brain has its own ledger at `<brain>/.claude/brain/invocation-
ledger.jsonl`. Claude Code sessions only append to the ledger for the
PROJECT ROOT they run in. A session at the ecosystem root writes to
ecosystem's ledger; a NeuroGrim-rooted session writes to NeuroGrim's.
The four Brains accrue their own separate usage data.
