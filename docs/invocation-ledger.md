# Invocation Ledger — Setup Guide

The invocation ledger captures every `Skill` tool call in a Claude Code
session and persists a one-line JSONL record per invocation to the
Brain's runtime state. The `capability-hygiene` domain reads the ledger
and classifies each skill as **alive**, **dead**, or **new** — closing
the empirical half of the three-axis self-observability picture
(hygiene · invocation · miss rate).

Axis 4 v1 (2026-04-22). Skills only. Subagents + tools are a future
extension once the v1 signal is validated in dogfood.

> **Correction 2026-04-22 (post-audit):** the `Skill` PostToolUse
> hook only fires for skills that are actually `Skill`-tool-invocable
> — i.e. skills structured as `.claude/skills/<name>/SKILL.md` with
> YAML frontmatter. Legacy Markdown skills at
> `.claude/skills/<name>.md` are **Read** by agents, not invoked via
> the `Skill` tool, and so are invisible to this ledger until
> migrated. Until the migration lands, the `capability-hygiene`
> domain will classify every legacy skill as `dead` once its 30-day
> grace period expires — this is an expected false positive, not a
> real signal. Tracked in BACKLOG under the Tier A migration; the
> sensor tooling already supports both formats, so migrated skills
> start producing real usage signal immediately.

## What gets captured (privacy-by-design)

Each invocation records:
- `schema_version`: `"1"`
- `ts`: ISO 8601 UTC timestamp (generated by the hook, not Claude Code)
- `type`: `"skill"` (v1) | `"disposition"` (v2) | `"pipeline"` (v3)
- `name`: the skill name (e.g., `rubber-duck`) or `<broker_id>/<pipeline_id>` (v3)
- `session_id`: Claude Code session UUID (for correlation only)
- `invocation_id`: the `tool_use_id` (unique per call)

**v3 additions for `type: "pipeline"` rows** (broker pipeline dispatches):
- `audit_class`: `"capability"` | `"governance"` | `"meta-observation"` — controls
  whether `capability-hygiene` reads the row (capability + governance: YES;
  meta-observation: NO — excluded from the feed they themselves consume; closes the
  self-referential loop where a pipeline observing the ledger inflates its own
  apparent aliveness). See [`BROKER-INTERNALS.md`](BROKER-INTERNALS.md) §2.3.

**NOT captured:** tool arguments, tool responses, transcript content,
user prompts, agent reasoning, pipeline parameters, pipeline outputs. The v1
invariant (name + timestamp only) extends verbatim to v3 (broker_id/pipeline_id +
timestamp + audit_class only). Nothing sensitive enters the ledger; no scrubbing
infrastructure is needed.

> **v2 (2026-04-27):** Disposition records (a sibling row kind for
> operator-calibration; LSP-Brains v2.11 §17.12) extend the same
> privacy invariant. See "Disposition Records" below. Free-text
> justification is structurally forbidden at v1; the schema's
> `additionalProperties: false` on `DispositionEntry` enforces this.

> **v3 (2026-06-23):** Broker pipeline dispatches are recorded alongside skill
> invocations under `type: "pipeline"`. The privacy-by-design invariant is preserved
> (name + timestamp + `audit_class` only; no parameters, no outputs). The
> `capability-hygiene` domain v-next folds pipelines into the same alive/dead/new
> classification namespace (skills + pipelines compete in the same scoring space). See
> [`BROKER-AWARENESS.md`](BROKER-AWARENESS.md) §3 for the broker-side rationale and
> [`BROKER-INTERNALS.md`](BROKER-INTERNALS.md) §2.3 for the `audit_class` semantics.
> The hook script for pipeline dispatches will ship alongside the broker framework
> code (post-S0-T); v1/v2 hooks coexist in the same hook config.

## Installation

1. **Confirm the hook script exists.** This repo ships
   `scripts/record-skill-invocation.sh`. It's a shell script using
   only bash builtins — no `jq`, no Python, no external tools.

2. **Add the PostToolUse hook to your local settings.** The hook
   lives in `.claude/settings.local.json` (operator-local, gitignored)
   at the **project root where you run Claude Code**. For the
   Brains ecosystem that's `D:/Brains/.claude/settings.local.json`.

   Merge this into the file, preserving any existing `permissions`
   or other sections:

   ```json
   {
     "hooks": {
       "PostToolUse": [
         {
           "matcher": "Skill",
           "hooks": [
             {
               "type": "command",
               "command": "bash \"$CLAUDE_PROJECT_DIR/NeuroGrim/scripts/record-skill-invocation.sh\""
             }
           ]
         }
       ]
     }
   }
   ```

   Adjust the path if your Claude Code session is rooted somewhere
   other than the ecosystem root. For a NeuroGrim-only session, use:
   ```
   "command": "bash \"$CLAUDE_PROJECT_DIR/scripts/record-skill-invocation.sh\""
   ```

3. **Restart Claude Code.** Settings are read at session start; a
   restart ensures the hook is active.

4. **Verify the hook fires.** Invoke any skill (e.g., via
   `/keybindings-help` or any other harness skill). Check that
   `.claude/brain/invocation-ledger.jsonl` now has a line. Parsed
   JSON should look like:

   ```json
   {"schema_version":"1","ts":"2026-04-22T11:20:43Z","type":"skill","name":"keybindings-help","session_id":"...","invocation_id":"..."}
   ```

## Consumed by `capability-hygiene`

On every `neurogrim sensory capability-hygiene` run, the sensor reads
the ledger and adds per-skill fields:

- `usage_status`: `alive` | `dead` | `new`
- `usage_rarity`: `common` (default) | `rare` (declared in skill
  frontmatter)
- `window_days`: 90 (common) or 365 (rare)
- `invocations_in_window`: count in the relevant window
- `last_invoked`: most-recent timestamp (RFC 3339) or `null`
- `skill_age_days`: file mtime age in days

Aggregate counts in `capability_breakdown.skills`:

- `usage:alive`, `usage:dead`, `usage:new` — per-status totals
- `ledger:total_invocations` — overall ledger size
- `ledger:low_confidence` — `1` if below
  `LOW_CONFIDENCE_TOTAL_INVOCATIONS` (20), `0` otherwise

## Dead-skill findings

Dead findings are **advisory** — they don't subtract points from the
hygiene score (hygiene measures description quality; usage is a
separate axis). They surface in the `findings` array as:

```
dead-skill:<name>                      — normal signal
low-confidence-dead-skill:<name>       — ledger too sparse to trust
```

**Plan-critic discipline for acting on dead findings:**

- Combine with hygiene score before acting:
  - *dead + hygiene < 60* → probably misdescribed (fix the lead
    paragraph before declaring the skill retired).
  - *dead + hygiene ≥ 90* → probably genuine niche (consider
    `usage-rarity: rare` or archive).
- Wait for `ledger:total_invocations` to exceed 20 before treating
  signals as firm. Below that, every finding is marked
  `low-confidence-*`.
- A file freshly created (mtime < 30 days) is `new`, not `dead` —
  grace period honors skills that haven't had time to accrue
  invocations.

## Troubleshooting

**Hook doesn't appear to fire** — Check:
1. `.claude/settings.local.json` syntax is valid JSON.
2. Path to the script is correct and the file is executable
   (`chmod +x`).
3. On Windows, Claude Code uses Git Bash — the shebang `#!/bin/bash`
   should work. `/dev/stdin` is NOT available on Git Bash (use `cat`
   without arguments to read stdin).
4. Claude Code was restarted after the settings edit.
5. You're in a fresh session — existing sessions may have cached
   settings.

**Hook exits 1 with "CLAUDE_PROJECT_DIR is unset"** — This is the
guardrail (concern C1 in the 2026-04-22 deep audit). Claude Code
normally populates `CLAUDE_PROJECT_DIR` automatically; if it's
missing, the hook is refusing to silently scatter ledger data into
whatever `$PWD` the bash process launched with. Check:
1. The hook was invoked via Claude Code (not run manually).
2. You're on a Claude Code build that sets `CLAUDE_PROJECT_DIR`.
3. If running the hook from a harness/test, set the env var
   explicitly.

**Ledger lines are malformed** — The capability-hygiene reader
tolerates malformed lines (silently skips them). If this is
systematic, check the hook script — the printf format string has to
escape quotes correctly.

**CMDB shows all skills as "new"** — This is expected right after a
git clone (all file mtimes are the clone time, so all skills are
inside the 30-day grace period). Within a month, skills will either
accrue invocations or age out of the grace period and start getting
flagged dead.

## Disposition Records (Axis 4 v2 — 2026-04-27)

Brains-2.0 E-B2-6 (LSP-Brains v2.11 §17.12) introduces a sibling row
kind on the same `.claude/brain/invocation-ledger.jsonl` file:
`DispositionEntry` rows that capture the operator's judgment of a
prior skill invocation. The new schema
`invocation-ledger-v1.schema.json` (LSP-Brains, v2.11) formalizes
both row kinds via a top-level `oneOf` discriminator.

**What gets captured on a disposition record:**
- `schema_version`: `"1"`
- `ts`: ISO 8601 UTC timestamp (generated by the disposition CLI)
- `entry_kind`: `"disposition"`
- `invocation_id`: references the `invocation_id` of an earlier
  `SkillEntry` row in the same ledger
- `disposition_kind`: closed-set 4-entry vocabulary —
  `accepted`, `rejected`, `modified`, or `superseded`
- `human_operator`: the operator handle (mirrors the
  `NEUROGRIM_OPERATOR` convention from §17.6)

**Privacy invariant preserved.** Disposition records honor the v1
contract: closed-set vocabulary + ts + invocation_id + operator
handle ONLY. **Free-text justification is FORBIDDEN at v1**; the
schema's `additionalProperties: false` on `DispositionEntry`
enforces this structurally. v2 may re-open with strict prose-only-
no-paths discipline + dedicated opt-in (BACKLOG B-23).

**Capture mechanism: explicit CLI only.**

```
$ NEUROGRIM_OPERATOR=alice neurogrim disposition record \
    --invocation-id <id-from-an-earlier-skill-row> \
    --kind accepted \
    --project-root .
```

Auto-inference of disposition from session traces (operator
immediately invoked another skill, edited a file, etc.) is OUT OF
SCOPE for v1 and is a v2 candidate per BACKLOG B-23. v1's
explicit-only posture is structural — observing follow-up Bash /
Edit / Write actions requires a PostToolUse-hook schema bump that
the existing `Skill`-only matcher does not cover.

**Recursion guard.** The disposition CLI rejects `--invocation-id`
references whose source is an `operator_calibration:*` finding kind
(spec §17.12.5). The `operator-calibration` sensor's own findings
are NOT eligible disposition targets — self-loop closed by
construction.

**Consumer.** A new sensor `neurogrim-sensory::operator_calibration`
reads the SAME ledger file and recognizes both row kinds. The
existing `read_invocation_ledger()` in `capability-hygiene` is
intentionally untouched — it silently skips DispositionEntry rows
(no `name` field at the skill level), so the two sensors operate
independently. No coupling.

**Aggregation-only export.** The operator-calibration sensor's CMDB
output emits aggregate totals only (per-disposition-kind counts +
sample-size denominators); it does NOT emit per-invocation rows or
per-skill breakdowns at v1. This is the BR-5 privacy mitigation;
relaxing it requires a charter-level conversation, not a Layer-2
lift.

**Sample-size disclosure.** The sensor reports `dispositioned_count`
AND `total_invocations` separately as exported variables. Score is
`null` when `dispositioned_count < N_MIN=20` (mirrors
`LOW_CONFIDENCE_TOTAL_INVOCATIONS` precedent in
`capability_hygiene.rs`). Below N_MIN, an
`operator_calibration:low_confidence` advisory finding fires.
Selection bias (operators disposition rare events; routine accept-
all use is invisible) is the dominant interpretive risk; the
sample-size denominator surface is the structural mitigation.

**Cross-Brain.** Disposition records stay LOCAL to each Brain
(mirrors the §17.8 lock for domain-calibration). No A2A
propagation in v1.

## Cross-Brain behavior

Each Brain has its own ledger at `<brain>/.claude/brain/invocation-
ledger.jsonl`. Claude Code sessions only append to the ledger for the
PROJECT ROOT they run in. A session at the ecosystem root writes to
ecosystem's ledger; a NeuroGrim-rooted session writes to NeuroGrim's.
The four Brains accrue their own separate usage data.
